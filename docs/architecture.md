# アーキテクチャ

## 全体構成

初期構成は以下のコンポーネントに分ける。

- RaceFront
- AdminFront
- API サーバー
- スクレイピングサーバー
- モデルトレーニングサーバー
- DB

## RaceFront

レース表示用と予測比較表示用のフロント。

React で作成し、まずは Web として提供する。将来的には native アプリ展開を見据える。

### 日付ごとのレース一覧ページ

表示要件:

- 開催場
- レース番号
- 結果確定または未確定
- レース詳細への遷移

機能:

- その日のレースデータ更新。
- 日付切り替え。

### レース詳細ページ

表示要件:

- 馬番
- 枠番
- 馬名
- 騎手名
- 馬体重
- 馬体重の増減
- 市場単勝オッズ
- 人気
- 予測単勝オッズ
- 予測確率
  - 1着率
  - 2着以内率
  - 3着以内率
- 疑似ROI
- おすすめ購入馬券
- 予測根拠

機能:

- レースデータの更新。
- 予測更新。

## AdminFront

特徴量データの更新やモデル作成用の管理フロント。

Web のみで作成する。

主な責務:

- 特徴量計算で累積統計が必要になる場合の更新と管理。
- データ取り込みの実行と結果確認。
- モデルトレーニングのオペレーション。
- モデル評価結果の確認。
- モデル採用、再校正、更新の操作。

## API サーバー

アプリケーションの中心 API。

主な責務:

- DB アクセス。
- RaceFront / AdminFront 向け API。
- 予測モデルを使用した予測実行。
- 疑似オッズ、疑似ROI、買い目候補の算出。
- モデルトレーニングサーバーへの仲介。
- スクレイピングサーバーへの仲介。

## スクレイピングサーバー

netkeiba スクレイピングロジックを持つ。

主な責務:

- レース一覧、出馬表、結果、オッズの取得。
- 馬、騎手、調教師情報の補完。
- Playwright による JavaScript ページの取得。
- 取得結果を DB に反映するための正規化。

## モデルトレーニングサーバー

モデル生成用の責務のみを持つ。

必要に応じて FastAPI などで API サーバーから呼び出す。

主なフェーズ:

- 特徴量生成。
- ハイパーパラメータ探索。
- トレーニング。
- 校正器作成。
- モデル精査。
- 採用判定。

初期モデルは LightGBM を使用する。

## DB

初期は `aiuma` の core schema を参考に、以下のテーブルを中心にする。

- `races`
- `horses`
- `jockeys`
- `trainers`
- `race_horses`
- `race_results`

その他の予測・推奨・モデル管理テーブルは後続設計で決める。
