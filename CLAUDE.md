## Codex agent の使用方針

**Proactively use codex** — 以下のトリガーに該当したら、ユーザーが明示的に頼まなくても
default で codex 系を起動する。「念のため」ではなく該当時は並走が既定動作。

### 目的

Codex は作業量削減ではなく、**判断の校正**のために使う。このリポジトリは
ML / features / training / serving / betting / ops / API が強く結合しており、
小さな変更でもリーク、校正悪化、prediction parity 破壊、DB/API 契約破壊、
backfill / ops job の冪等性破壊につながる。

高リスク変更では、実装後ではなく作業開始直後・方針確定前に Codex を
second opinion として呼び、Claude の暫定方針に穴がないか確認する。
Codex は意思決定者ではない。最終判断は Claude が行い、既存 spec、
constitution、テスト、実 DB E2E の制約と照合して採否を決める。

### 起動トリガー（1 つでも該当したら使う）

**ユーザー発話キーワード**（second opinion 系）:
「設計どう思う」「2 案出して」「これで合ってる？」「比較して」「レビューして」
「アーキ」「方針」「相談」「迷ってる」「どっち」「妥当？」
→ `codex:codex-rescue` agent を Agent tool で並走させ意見を取る

**作業内容（MUST）**:
- 新規 spec / plan / 設計ドキュメントを書く / 更新する
- 新規 API / service / module / CLI entrypoint / ops job を設計する
- DB migration / schema / OpenAPI / 永続化形式 / artifact manifest を変更する
- 複数パッケージ横断の refactor / migration / schema 変更を行う
- features / training / eval / calibration / serving / betting のロジックを変更する
- materialized features / fingerprint / parity / fail-closed / backfill / idempotency に触る
- 同一バグで 1 回直して再発、または root cause が読めない

**このリポジトリ固有の MUST 条件**:
- as-of / strictly-before / same-day exclusion / 自馬除外 / market odds 非特徴化に触る
- OOS 評価、fold、採用ゲート、ECE、calibration、feature ablation に触る
- training / serving / eval の特徴量経路一致に触る
- `FEATURE_VERSION`、model artifact、feature hash、manifest、parquet materialization に触る
- prediction parity / bit parity / dtype drift / fail-closed に触る
- recommendation / Kelly / ROI / pseudo odds / real odds / backtest 表示に触る
- admin / ops から書き込みアクションを増やす

**スタック時の追加ルール**:
「もう一回試す」前に必ず codex を呼ぶ。再試行 2 回目は禁止。

### Codex に確認させること

Codex には原則として実装を依頼しない。まず以下を確認させる。

- 設計の穴
- 代替案
- 主要リスク
- 必要なテスト
- 見落としそうな既存制約
- このリポジトリ固有のリーク・パリティ・契約・冪等性リスク

### 依頼テンプレート

Codex を呼ぶときは、最低限以下を渡す。

- 変更目的
- 触る予定のファイル・パッケージ
- Claude の暫定方針
- 守るべき既存制約
- 確認してほしい観点
- 「実装は不要。レビューと代替案だけ返す」

### 結果の扱い

- codex 結果は鵜呑みにせず Claude 案と突き合わせる
- 採用する指摘、採用しない指摘と理由、保留する論点を明示する
- 追加するテスト、または追加しない理由を明示する
- 残リスクがあれば明示する
- Codex の指摘が 0 件でも、「差分比較した結果、追加変更なし」と明示する
- Codex を呼んだ事実だけで完了扱いにしない

### 使わない場面（狭めに定義）

- 1 ファイル内の rename / typo / import 修正
- 既に固まった方針の機械的な反映
- テスト期待値だけの明白な更新
- ドキュメントの表記ゆれ修正
- ユーザーが明示的に「Claude だけで」「Codex なしで」と指示

### 実行ルール

- 並走時は Agent tool を単一メッセージから並列発火（Claude 自身の作業と同時進行）
- 初回は `/codex:setup` で CLI 稼働を確認
- 起動を見送った場合、応答の冒頭で「codex 不要と判断した理由」を 1 行宣言する
  （これが最大の効き目 — 見送り判断が可視化される）

### Codex 起動失敗時

- 初回失敗時は `/codex:setup` または環境確認を行う
- 同一タスクでの再試行は最大 1 回
- 失敗した場合は `codex unavailable: 理由` を作業ログまたは spec summary に残す
- 高リスク変更で Codex が使えない場合、Claude は代替セルフレビュー checklist を実施する
- 連続して失敗する場合、機能実装ではなく Codex 環境復旧を別タスク化する
