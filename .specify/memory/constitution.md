# 競馬予測システム (Horse Racing Prediction) Constitution

本 Constitution は、Obsidian Vault (`horseracing/`) 由来の docs を上位の不変条件として
成文化する。docs と個別 spec が本憲法に矛盾する場合は、本憲法を優先する。

## Core Principles

### I. データ契約の固定 (ID & スキーマ)
レース・馬・騎手・調教師を横断するすべての処理は、固定された識別子契約に従わなければ
ならない。`raceId` は `^[0-9]{12}$` を満たす 12 桁固定文字列 (`YYYYVVKKDDRR`) とし、
日付ではなく識別子として扱う。JRA-VAN データは 2007 年以降のみを取込・学習・評価に
使用し、2006 年以前は ID 体系が異なるため使用してはならない。JRA-VAN と netkeiba の
馬・騎手・調教師 ID は、`id_mappings` による対応付けなしに横断利用してはならない。
対応が取れない場合は手動修正フローに載せ、推測で結合してはならない。予測ラベル名は
`1着率` / `2着以内率` / `3着以内率` を正とし、`win` / `place` / `show` などの英語名は
実装内部の識別子としてのみ使用する。理由: ID とラベルの契約が揺れると、リーク・誤結合・
評価不能を招き、全工程の信頼性が崩れる。

### II. リーク防止 (NON-NEGOTIABLE)
予測時点で利用不能な情報を、特徴量・予測入力に混入させてはならない。すべての特徴量は
`source`・利用可能タイミング (出馬表前 / 枠順後 / 馬体重後 / オッズ後 / 直前 / 結果後)・
欠損処理を必須記載する。結果確定後の情報は学習ラベルと評価にのみ使用でき、未来レース
予測の特徴量に使ってはならない。累積統計・カテゴリ統計・血統 embedding は、対象レース
より前のデータのみ、または train-only / out-of-fold で計算し、評価期間の結果を混入させ
てはならない。学習・評価は walk-forward を日時で分割し、分割境界日を特徴量計算に必ず
適用する。市場オッズは初期方針では予測モデルの入力特徴量に使わない。特徴量化を試す
場合は、別 spec でリーク防止・利用可能タイミング・評価方法を定義してからとする。理由:
リークを含む精度は運用価値を持たず、長期回収率を毀損する。

### III. 評価先行 (NON-NEGOTIABLE)
モデルまたは特徴量を変更する feature は、walk-forward out-of-sample 評価を必須とする。
モデル採用は、同一評価条件での baseline (人気順など) 比較と確率校正 (ECE) 確認を必須と
し、これらなしに採用判定してはならない。予測品質指標 (LogLoss / Brier / AUC / NDCG /
ECE) と運用指標 (疑似ROI / 回収率 / 的中率 / 最大ドローダウン / 最大連敗数 / 見送り率)
を計測できる評価ハーネスを、学習ロジックより先に用意する。理由: 評価基盤なしの学習は
採用判断を不能にし、改善か劣化かを検証できないまま実装コストを積み増す。

### IV. 確率整合性
各馬の出力確率は `0 <= 1着率 <= 2着以内率 <= 3着以内率 <= 1` を満たさなければならない。
レース内合計は `Σ1着率 ≈ 1`、`Σ2着以内率 ≈ 2`、`Σ3着以内率 ≈ 3` に近いこと。許容誤差は
spec で定義する。出走取消・競走除外が確定した馬は、予測母集団・正規化・券種計算・
疑似ROI・買い目候補から除外し、残りの出走馬で再正規化する (画面では状態のみ表示して
よい)。過去成績のない馬 (新馬等) は固定特徴量スキーマを保ったまま欠損 (Unknown / null)
として渡し、存在しない過去成績に `0` を代入してはならない (Unknown と 0 を区別する)。
理由: 馬券戦略の基礎は全頭確率の一貫した品質であり、整合性違反は券種確率と疑似ROIを
無意味にする。

### V. 再現性と監査
予測・推奨を保存する場合、`model_version`、特徴量定義版、予測実行記録、計算時に使用した
市場オッズ / 推定市場オッズ / 疑似オッズ / 疑似ROI、計算時刻、ロジックバージョンを結果
レコードに保持しなければならない。オッズはスナップショット履歴を保存せず、最新値で
上書きし `updated_at` のみ保持する。提示済みの判断結果は上記の監査情報で後から確認できる
ようにする。推定市場オッズを用いた評価・ROI は「疑似評価」として明示し、実オッズ由来の
値と区別する。理由: 最新オッズ上書き方針の下でも、提示済み判断の事後検証と改善比較には
最小限の監査情報が不可欠である。

### VI. feature 分割規律 (spec-kit 運用)
UI を持つ feature は、先に API / DB の契約を確定してから着手し、契約なしに画面を作っては
ならない。P0 未決事項 (結合確率エンジン、推定市場オッズ変換規則、予測・推奨テーブル) を
含む feature は、実装前に spec / plan で方式を固定する。予測・推奨系テーブル
(`model_versions` / `prediction_runs` / `race_predictions` / `feature_snapshots` /
`id_mappings` / `ingestion_jobs`) の最小契約は、初期 DB 設計の段階で含め、後付けの作り直し
を避ける。理由: 契約を欠いた先行実装は API と画面の手戻りを生み、spec-driven の利点を失う。

## 技術・スコープ制約

- 初期はデータ取込・特徴量更新・学習・予測・モデル更新をすべて手動実行とする。自動運用
  (スケジューラ / 各種検知 / 定期再学習) は将来スコープとする。
- 初期モデルは LightGBM を使用する。血統 embedding は主モデルではなく補助特徴量として
  扱う。
- 5 コンポーネント (RaceFront / AdminFront / API / スクレイピング / 学習) は将来の論理
  責務であり、初期から独立サービスに分割することを強制しない。スクレイピング・学習は
  当面スクリプト / CLI もしくは API 経由実行でよい。サービス分割は自動運用フェーズで必要
  になってから行う。
- DB の初期スキーマは `aiuma` の core schema (`races` / `horses` / `jockeys` / `trainers` /
  `race_horses` / `race_results`) を参考にする。

## 実装順の原則

- ML コア (評価ハーネス → 特徴量 → 学習 → 校正) を、netkeiba スクレイピングおよび本格的な
  フロント実装より前に、過去 JRA-VAN データのみで baseline 超えを検証する。
- netkeiba スクレイピングは未来レース予測の serving 段階まで後ろ倒ししてよい。

## 開発・レビュー品質ゲート

- すべての implementation plan は Constitution Check で、`raceId` 契約、リーク防止、時系列
  split、全頭 `1着率` / `2着以内率` / `3着以内率`、確率整合性、評価先行 (baseline + ECE)、
  予測・推奨テーブル契約、疑似オッズ / 市場比較、再現性・監査を検証しなければならない。
- すべての tasks は、機能に応じて leakage test、時系列 split test、確率整合性 test、評価
  ハーネス test、疑似オッズ / 市場オッズ比較 test、推奨根拠監査 test を含めなければ
  ならない。
- speckit の plan / tasks / implement フェーズで、アーキテクチャ選定、特徴量設計、確率推定
  方式、推奨ロジックなど非自明な設計判断を行う場合、`codex:codex-rescue` agent または
  `/codex:rescue` skill に second opinion を求め、両案の差分と採用根拠を plan.md または
  tasks.md に記録しなければならない。単純な edit / rename / 既存パターンの踏襲、および明確に
  方針が固まった軽微な変更はこの限りでない。理由: 単一視点の設計判断は重要なトレード
  オフを見落としやすく、回収率・確率品質・運用再現性に長期的なリスクを残すため。

## Governance

本 Constitution は、特徴量設計、学習・評価、推奨ロジック、運用 UI を含むすべての docs・
spec・plan・tasks・レビュー判断に優先する。spec / plan / 実装が本憲法に反する場合は、
本憲法を優先する。すべての plan・実装レビューでは、原則 II (リーク防止)・III (評価先行)・
IV (確率整合性) の遵守を必ず検証しなければならない。改定は、変更理由・影響範囲・関連
テンプレート更新・必要なら移行方針を同一変更セットで提示した場合にのみ認める。バージョン
管理は semantic versioning を用い、新原則や必須ゲートの追加は MINOR、既存原則の後方互換性
を壊す変更は MAJOR、文言明確化のみは PATCH とする。違反がある場合は plan / spec / tasks の
修正か、明示的な憲法改定のいずれかを要求する。
