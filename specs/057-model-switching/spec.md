# Feature Specification: 複数モデル切り替え基盤(用途ラベル + レース詳細でのモデル切替)

**Feature Branch**: `057-model-switching`

**Created**: 2026-07-06

**Status**: Implemented(2026-07-06、worktree `057-model-switching`。全スイート緑: db +1 / api 101(+8)/ training set-model-label +3 / front 74(+5)/ admin 22(+3)、ruff/tsc/eslint/drift クリーン、head-assert 0011。プロダクト DB への migration 0011 適用は deploy 手順で未適用)

**Input**: User description: 複数の用途ラベル付きモデルを共存させ、レース詳細画面で予測に使うモデルを切り替えられるようにする。デフォルトは採用モデル(active)。将来の B1(過去走オッズ履歴特徴の「精度最優先」モデル)を意思決定支援モデルと別用途で共存・比較するための土台。

## 背景と目的 *(non-normative)*

現在、予測 API は 1 レースに対し常に「採用モデル(`adoption_status='active'`)」の予測 run のみを返す。予測は既に (レース × モデル) 単位で永続化されており(`prediction_runs` / `race_predictions`)、`serving predict-backfill --model-version` で複数モデルの予測を同一 DB に共存させる仕組みも既存(044)。不足しているのは、(a) 各モデルが「何のためのモデルか」を人間が判別できる用途情報、(b) 予測画面で採用モデル以外の予測を明示的に選んで見る手段、の 2 点のみ。

このフィーチャは切り替え「基盤」であり、精度最優先モデル(B1: 過去走オッズを履歴特徴にしたモデル)そのものは実装しない(後続 feature)。B1 のような市場オッズ由来特徴の追加は憲法 II により別 spec でリーク防止・利用可能タイミング・評価方法を定義してから行う。本 feature は特徴量・モデル学習・リーク境界に一切触れない、表示と serving 選択の配管である。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - モデルの用途がわかる(Priority: P1)

オペレータがモデル一覧(admin レジストリ)や予測画面で各モデルを見たとき、`lgbm-055` のような技術 ID だけでなく「何のためのモデルか(例: 意思決定支援=市場独立予測 / 精度最優先)」が人間可読で分かる。

**Why this priority**: 複数モデルが共存し始めた瞬間、用途が分からなければ切替も比較も意味を持たない。切替 UI(US3)の前提。技術 ID は保ちつつ用途情報を追記するだけで、単独でも「レジストリで各モデルの目的が読める」という価値を出す。

**Independent Test**: モデルに用途ラベルを設定 → `/api/v1/models` と admin レジストリでラベル/用途が表示されることを確認。予測切替 UI が無くても完結してテスト可能。

**Acceptance Scenarios**:

1. **Given** 複数のモデルが登録されている, **When** モデル一覧を取得する, **Then** 各モデルに技術 ID(`model_version`)に加え人間可読ラベルと用途説明が付いて返る。
2. **Given** 用途情報が未設定の既存モデル, **When** モデル一覧を取得する, **Then** ラベル/用途は未設定(null)として返り、エラーにならず技術 ID で識別できる。
3. **Given** モデル `model_version`(技術 ID), **When** 用途情報を追加する, **Then** `model_version` 自体は変更されず(リネームされず)、既存の予測 run・artifact・logic_version の参照が壊れない。

---

### User Story 2 - 予測を任意モデルで取得できる(Priority: P1)

予測画面(または API 利用者)が、特定のモデルを指定してそのレースの予測を取得できる。指定しなければ従来どおり採用モデルの予測が返る。

**Why this priority**: 切替の本体。省略時=採用モデルという後方互換を保ちつつ、モデル指定という選択肢を追加する。US3 の UI はこの API を呼ぶだけになる。

**Independent Test**: あるレースに対しモデル A・B の予測 run を両方永続化 → モデル未指定で採用モデルの予測、モデル B 指定で B の予測が返ることを確認。UI 無しで API 単体テスト可能。

**Acceptance Scenarios**:

1. **Given** あるレースに採用モデルの予測 run が存在, **When** モデルを指定せず予測を取得, **Then** 従来と完全に同一の応答(採用モデルの予測)が返る。
2. **Given** あるレースに非採用モデル B の予測 run が存在, **When** モデル B を指定して予測を取得, **Then** B の予測(B の run の各馬確率・監査情報)が返る。
3. **Given** あるレースにモデル C の予測 run が存在しない, **When** モデル C を指定して予測を取得, **Then** 型付きエラー(「そのモデルはこのレースを予測していない」)が返り、黙って採用モデルにフォールバックしない。
4. **Given** 存在しない model_version を指定, **When** 予測を取得, **Then** 型付きエラーが返り、未処理の 500 にならない。

---

### User Story 3 - レース詳細で予測モデルを切り替える(Priority: P2)

レース詳細画面で、予測に使うモデルを選択できる。初期表示は採用モデル。選択を変えると、その馬ごとの予測がそのモデルのもので再表示される。

**Why this priority**: US1/US2 を人が使える形にする層。UI は US2 の API を呼ぶだけなので、API が正しければ薄い。P2 は US1/US2(バックエンド契約)が先に固まる必要があるため。

**Independent Test**: レース詳細を開く → 既定で採用モデルの予測が採用バッジ付きで表示 → セレクタで別モデルを選ぶと予測が切り替わる → そのモデルの予測が無いレースでは「未生成」状態が出る、を UI テストで確認。

**Acceptance Scenarios**:

1. **Given** レース詳細画面, **When** 画面を開く, **Then** 採用モデルの予測が表示され、採用モデルであることが視覚的に分かる(バッジ等)。
2. **Given** 複数の選択可能モデル, **When** 別モデルを選択, **Then** 予測がそのモデルのもので再取得・再表示され、どのモデルを見ているかが常に画面に明示される。
3. **Given** 選択したモデルの予測がこのレースに無い, **When** そのモデルを選ぶ, **Then** ローディング/空/一般エラーとは区別された「このモデルはこのレースを予測していない(未生成)」状態が表示される。

---

### Edge Cases

- 選択したモデルの予測がそのレースに存在しない → 型付き「未生成」状態(404 相当)。採用モデルへの暗黙フォールバックはしない(見ているモデルの誤認を防ぐ)。
- 用途ラベル/説明が未設定の既存モデル → null 表示(空文字や 0 埋めをしない)。技術 ID で必ず識別可能。
- 採用モデルが 1 つも無い(active 不在)→ 既存挙動を維持(予測未生成の型付き空応答)。モデル指定時はそのモデルの run 有無のみで判定。
- 同一モデルにそのレースの run が複数(再生成)→ 最新(computed_at DESC → run_id DESC)を選ぶ、採用モデル選択と同じ決定規則。
- モデル選択パラメータ省略 → 従来と完全に同一の応答(後方互換)。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは各モデルに、技術 ID(`model_version`)とは別に、人間可読の用途ラベルと用途説明を保持できなければならない。技術 ID は不変とし、用途情報の追加によって既存の予測 run・artifact・logic_version の参照を壊してはならない(リネーム禁止・追記のみ)。
- **FR-002**: モデル一覧(`/api/v1/models`)と admin レジストリは、各モデルの用途ラベル・用途説明を透過表示しなければならない。未設定は null として返し、エラーにしてはならない。
- **FR-003**: 予測取得は、モデルを指定する任意のパラメータを受け付けなければならない。パラメータ省略時は従来どおり採用モデル(active → 最新 run)の予測を返し、既存の応答と完全に同一でなければならない(後方互換)。
- **FR-004**: モデルが指定された場合、システムはそのモデルの予測 run を決定的に選択(そのモデルの run を computed_at DESC → run_id DESC)して返さなければならない。採用モデルかどうかは選択条件に影響しない。
- **FR-005**: 指定されたモデルにそのレースの予測 run が存在しない場合、システムは型付きエラー(「そのモデルはこのレースを予測していない」)を返さなければならず、採用モデルへ暗黙にフォールバックしてはならない。存在しない model_version 指定も型付きエラーとし、未処理例外(500)にしてはならない。
- **FR-006**: レース詳細画面は予測モデルの選択手段を提供し、初期選択を採用モデルとしなければならない。どのモデルの予測を見ているかを画面に常時明示し、採用モデルを視覚的に区別しなければならない。
- **FR-007**: 予測画面は「選択モデルの予測が未生成」を、ローディング・空・一般エラーと区別した状態として表示しなければならない。
- **FR-008**: システムは予測を都度計算してはならない(API は read-only を維持し、モデルを実行しない)。切替は永続化済みの予測 run の読み替えのみで行う。新規モデルの予測生成は既存の明示操作(predict-backfill / ops ジョブ)に限る。
- **FR-009**: 「採用(active=デフォルト表示)」と「評価に合格した(技術的に良い)」は独立した概念として扱わなければならない。あるモデルが評価に合格しても、それだけで自動的に採用(デフォルト表示)へ昇格させてはならない。レジストリは用途と選択可能性を単一の active フラグと独立に表現できなければならない。
- **FR-010**: 本 feature は特徴量定義・モデル学習・リーク境界・FEATURE_VERSION を変更してはならない。API 契約変更は純追加とし、省略時挙動の後方互換と read-only(全 path GET)を維持しなければならない。

### Key Entities *(include if feature involves data)*

- **モデル(model version)**: 予測を生成した学習済みモデル。技術 ID(不変)+ 採用状態 + (本 feature で追加)人間可読の用途ラベル・用途説明を持つ。用途情報はメトリクスではなくモデルの素性。
- **予測 run**: あるモデルがあるレースに対して生成し永続化した予測の 1 単位。(レース × モデル)で複数共存しうる。監査情報(model_version・logic_version・計算時刻)を持つ。
- **予測応答**: 1 レースの各馬確率(1着/2着以内/3着以内)+ どのモデル・run から来たかの監査情報。モデル選択パラメータに応じて対象 run が変わる。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: オペレータがモデル一覧を見たとき、各モデルの用途が技術 ID を知らなくても言葉で判別できる(用途未設定モデルは「未設定」と分かる)。
- **SC-002**: モデル未指定の予測取得は、本 feature 導入前と 100% 同一の応答を返す(既存の全 API テストが無改修で緑)。
- **SC-003**: あるレースに 2 つ以上のモデルの予測が永続化されているとき、オペレータはレース詳細画面上でそれらを切り替えて、どのモデルの予測かを常に把握しながら閲覧できる。
- **SC-004**: 選択したモデルの予測が無いレースで、オペレータは「未生成」であることを、障害・空データ・読み込み中と混同せずに認識できる。
- **SC-005**: API 契約は純追加で、front/admin の OpenAPI drift-check が緑を維持し、read-only(全 path GET のみ)不変が保たれる。

## Assumptions

- 複数モデルの予測共存は既存インフラ(`prediction_runs` の (レース×モデル) 粒度、`serving predict-backfill --model-version` の冪等生成、044)で成立済み。本 feature は生成手段を追加しない。
- 切替 UI はエンドユーザー向けフロント(front)のレース詳細に置く(admin ではなく)。単一オペレータ・localhost 運用を前提(admin プログラム 051 方針)。認証はスコープ外。
- 用途ラベル/説明は `model_versions` に nullable な追加項目として保持する(migration = 現 head **0010**(056 raw_column_features)以降の追加 = **0011**、憲法 VI で正当化)。既存モデル行への遡及書換はしない(040 importance / 050 train_through と同じく、以後の設定・学習から populate)。
- 予測 run 選択の決定規則(computed_at DESC → run_id DESC)は採用モデル選択と一貫させ、モデル指定時は active 条件のみ外した形にする。
- **Deferred(本 feature 外)**: 精度最優先モデル B1(過去走オッズ履歴特徴)の実装そのもの — 憲法 II に従い別 spec。admin 側からのモデル切替。モデル間指標の横並び比較 UI・複数 run 時系列比較。認証。切替 UI からの予測生成(backfill)起動。
- **Codex**: 環境未インストールで本セッション 2 回起動失敗 → codex unavailable。single-opinion + セルフレビュー checklist で進める(横断 refactor・migration・API 契約変更のため本来 MUST-codex 案件)。
