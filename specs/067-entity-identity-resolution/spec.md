# Feature Specification: Entity Identity Resolution & Split Repair

**Feature Branch**: `067-entity-identity-resolution`

**Created**: 2026-07-12

**Status**: Draft

**Input**: User description: netkeiba サロゲートID(`nk:`)による馬・騎手・調教師レコードの分裂を、identity 解決(未マッピング→正規IDへの昇格)と既存分裂の物理 repair で統合し、トレーニング/予測の as-of 履歴切れ(silent degradation)を解消する。

## Context / Problem *(background)*

netkeiba スクレイプは、JRA-VAN 正規IDに紐付かないエンティティ(馬/騎手/調教師)を `nk:{netkeiba_id}` のサロゲートIDで**別レコードとして**取り込み、`id_mappings` に `mapping_status=unmapped` で記録する(「推測結合しない」設計)。

実 DB 調査により、この設計が現実のデータと噛み合っていないことが判明した:

- netkeiba の `source_id` の多くが JRA-VAN の正規IDと**同一採番**である。
  - 馬: サロゲート 9,647 頭中 **5,977 頭(62%)** で `nk:` の数字部分が既存 `jra_van` の `horse_id` と完全一致。うち馬名一致 5,977/5,977、生年一致 5,975。
  - 騎手: 176 中 163(93%)が番号一致。調教師: 220 中 207(94%)が番号一致。
- JRA-VAN フィードは 2025 年半ばで停止しており、**2026 年のレースは 100% netkeiba 由来**。2024 年以前にも走った現役の馬・騎手・調教師は軒並み 2 つのIDに分裂している。
  - 例: 馬「サヴォーナ」が `2020100734`(jra_van, 17 走 2022–2024)と `nk:2020100734`(netkeiba, 2 走 2026)に分裂。

**なぜ重大か**: 特徴量は `horse_id`/`jockey_id`/`trainer_id` をキーに as-of 集約(strictly-before)する。分裂により、netkeiba 側の直近レース(=まさに予測したいレース)を予測する際、過去走履歴が空になり「デビュー/低履歴」の誤った特徴で予測される。これは live/直近予測の **silent degradation** であり、表示上の不便に留まらない。騎手/調教師は human_form・target encoding(本リポジトリ最大の精度改善レバー)の入力なので同様に劣化する。血統の自馬除外(sire 累積−自馬)も 2ID 分裂で二重計上の軽微なリークを生む。

現状、`id_mappings` を `unmapped→mapped` に昇格させる production コードは存在せず(resolver/CLI/backfill 皆無)、resolver は `horses.horse_id` とも名前とも一切照合しない。features/serving/api はすべて生の ID 文字列で group/join し `id_mappings` を辿らないため、分裂が全経路に貫通している。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 既存分裂の統合で予測の履歴切れを解消する (Priority: P1)

運用者として、既に分裂している馬・騎手・調教師レコードを正規IDへ統合し、直近レースの予測が正しい過去走履歴に基づいて計算されるようにしたい。

**Why this priority**: これが本 feature の中核価値。2026 年の全レースが netkeiba 由来で現役馬が軒並み分裂しているため、統合しない限り現在の予測は「デビュー扱い」の誤特徴で汚染され続ける。トレーニング/予測に直結する唯一の根本解決。

**Independent Test**: サヴォーナ型の分裂ペア(過去 17 走が正規ID、直近 2 走がサロゲート)を含む DB に対して repair を実行し、統合後に直近レースの as-of 特徴が「過去 17 走を反映(デビュー扱いでない)」になること、馬詳細ページが 19 走すべてを一体で表示することを確認する。

**Acceptance Scenarios**:

1. **Given** 馬が正規ID(過去走)とサロゲートID(直近走)に分裂している、**When** repair を実行する、**Then** サロゲート側の出走/結果/予測行が正規IDへ寄せ替えられ、サロゲートのマスタ行は除去され、以後その馬の全履歴が単一IDに統合される。
2. **Given** 統合済みの馬が直近レースに出走している、**When** その直近レースの予測特徴を再計算する、**Then** 過去走履歴が空でなく実際のキャリアを反映する。
3. **Given** repair を一度実行済み、**When** repair を再実行する、**Then** 何も変更されない(冪等)。
4. **Given** 番号一致するが正規ID側に同一レースの行が既に存在する(衝突)、**When** repair を実行する、**Then** その行は寄せ替えずスキップし、衝突として報告する(既存の正規データを上書きしない)。

---

### User Story 2 - 今後の取り込みで分裂を発生させない (Priority: P1)

運用者として、以後スクレイプする馬・騎手・調教師が、既に正規IDが存在する個体なら最初から正規IDで取り込まれ、新たなサロゲート分裂を生まないようにしたい。

**Why this priority**: 出血源を止めなければ repair 後も再び分裂が積み上がる。P1 の repair と同格で、両輪。

**Independent Test**: 既存の正規IDと同一採番かつ同名の個体をスクレイプ取り込みし、サロゲート行が新規作成されず正規IDに解決されることを確認する。

**Acceptance Scenarios**:

1. **Given** 正規ID `X` が既に存在する、**When** `source_id=X` かつ同名の個体を取り込む、**Then** 正規ID `X` に解決され、`nk:X` のマスタ行は作られない。
2. **Given** 正規IDに対応が存在しない真に新規の個体(地方/新馬)、**When** 取り込む、**Then** 従来どおりサロゲートIDで取り込まれ、`unmapped` として記録される(誤統合しない)。

---

### User Story 3 - identity 解決の監査と安全な適用範囲 (Priority: P2)

運用者として、どのサロゲートがどの根拠で正規IDへ解決されたか、また自動解決を保留した曖昧ケースが何かを監査・確認できるようにしたい。

**Why this priority**: 「推測結合しない」原則を保ちつつ自動統合を正当化するには、解決根拠(番号一致+名前一致)と保留(生年不一致等)を記録・可視化する必要がある。中核価値ではないが安全な運用の前提。

**Independent Test**: identity 解決を実行し、mapped 化された各行に解決根拠が記録され、番号一致だが属性不一致のケースが自動解決されず「要確認」として区別されることを確認する。

**Acceptance Scenarios**:

1. **Given** サロゲートが正規IDと番号一致かつ名前一致、**When** identity 解決を実行、**Then** そのマッピングは根拠付きで `mapped` に昇格する。
2. **Given** サロゲートが番号一致だが生年不一致(馬 2 頭)、**When** identity 解決を実行、**Then** 自動解決されず「要確認(conflict)」として記録され、寄せ替えの対象外となる。
3. **Given** repair 前に何が寄せ替えられるかを確認したい、**When** dry-run で実行、**Then** DB を変更せず影響範囲(統合対象数・衝突・保留)のみを報告する。

---

### Edge Cases

- **番号一致・名前不一致**: 名前が一致しなければ自動解決しない(名前だけの推測結合も、番号だけの一致も、単独では根拠にしない)。
- **生年不一致(馬 2 頭)**: 番号・名前一致でも生年が異なる場合は conflict として保留(手動レビュー行き)。
- **番号不一致(残り 38%)**: 正規IDに対応が無い真の未マッピング(地方競馬/新馬等)は `unmapped` のまま。統合対象にしない。
- **衝突**: 寄せ替え先 `(race_id, canonical_id)` の行が既に存在する場合はスキップして報告(現状 0 件だが防御的に必須)。
- **買い目の JSON 内 ID**: 推奨(recommendations)は選択内容に馬IDを JSON で保持しており、ID 列の寄せ替えでは自動更新されない。統合後は該当レースの買い目を再生成して整合させる。
- **旧サロゲートURLの参照**: 統合でサロゲートのマスタ行が消えた後、旧サロゲートIDを指す詳細ページ参照が解決できること(正規IDへ誘導、または明示的な「未収録」応答)。
- **中断・部分実行**: repair が途中で中断しても、再実行で残りだけが処理され二重寄せ替えが起きないこと。

## Requirements *(mandatory)*

### Functional Requirements

**identity 解決(US2/US3)**

- **FR-001**: システムは、netkeiba サロゲートの `source_id` が既存の canonical ID(馬/騎手/調教師)と一致し**かつ**マスタの名前が一致する場合にのみ、当該マッピングを canonical ID へ解決(`mapped` 化)MUST。
- **FR-002**: システムは、番号一致だが名前不一致、または名前一致だが属性(生年など)不一致のケースを**自動解決してはならない**(保留/要確認として区別)MUST NOT。
- **FR-003**: システムは、正規IDに対応が存在しないサロゲート(真の未マッピング)を `unmapped` のまま維持 MUST(誤統合しない)。
- **FR-004**: identity 解決は馬・騎手・調教師に**共通(entity 汎用)**で適用可能 MUST。
- **FR-005**: 各解決について、解決根拠(番号一致+名前一致)と解決時刻を監査可能な形で記録 MUST。

**取り込み時の分裂防止(US2)**

- **FR-006**: 取り込み処理は、対象個体に対応する canonical ID が既存し**かつ照合に必要な情報(名前、馬は生年)が利用可能で一致する**場合、サロゲートIDを新規作成せず canonical ID を用いる MUST。
- **FR-006a**: 照合情報が取込経路で欠損する場合(結果のみ取込=ID のみ、血統親=ID のみ 等)は自動昇格せず、従来どおりサロゲートとして扱う MUST(欠損を誤って統合しない)。馬の生年は年齢とレース日から導出する規則を明文化・検証する。
- **FR-007**: canonical ID が存在しない個体は従来どおりサロゲートで取り込み、`unmapped` として記録 MUST(後方互換=既存の未マッピング動作を壊さない)。照合情報を追加する既存呼び出しは、情報省略時に従来動作を維持 MUST。

**既存分裂の repair(US1)**

- **FR-008**: システムは、解決済み(`mapped`)のサロゲートに紐づく、ID を保持する全データ(出走・結果・予測・特徴スナップショット)を canonical ID へ寄せ替え MUST。
- **FR-009**: repair は**冪等** MUST(再実行しても追加変更を生まない)。
- **FR-010**: repair は寄せ替え前に、寄せ替え先の主キーが既存でないことを検証し、既存(衝突)の場合はスキップして報告 MUST(既存の正規データを上書きしない=JRA-VAN 結果 INSERT-ONLY 原則の維持)。
- **FR-011**: repair は **dry-run モード**を備え、DB を変更せず影響範囲(統合件数・衝突・保留)を報告できる MUST。
- **FR-012**: 寄せ替え後、孤児となったサロゲートのマスタ行を除去 MUST。ただし `id_mappings` の解決記録は保持 MUST(監査可能性)。
- **FR-013**: 買い目(推奨)の選択内容(JSON)に含まれる馬IDは、過去の推奨がバックテストで結果と照合されるため、canonical ID へ**物理的に更新**MUST(再生成だけでは旧行のIDが不整合のまま残るため)。
- **FR-013a**: 派生データ(予測・特徴スナップショット)の「再生成」は既存処理が追記型(新実行を追加し既存を上書きしない)であることを前提とし、統合後の正しい値は**新しい実行**として提供され、統合前の実行は監査記録として区別可能に残す MUST。
- **FR-013b**: マスタ個体の削除前に、canonical 側で欠損し surrogate 側にのみ存在する属性(性別・生年・血統・馬主等)が無いことを検証 MUST(属性の取りこぼしによる情報損失を防ぐ)。血統の親ID参照(生ID文字列)に残る surrogate ID も canonical へ更新 MUST。
- **FR-013c**: repair は 1 個体ペア(surrogate→canonical)を不可分な単位として処理し、そのペアのいずれかの表で衝突・失敗が起きた場合はペア全体を無変更で見送る MUST(部分統合を作らない)。
- **FR-013d**: repair の実行記録(対象期間・件数・衝突・ツール版)を既存の取込ジョブ監査に永続化 MUST(実行後に何が変わったかを後から確認できる)。

**repair 後の後処理(US1)**

- **FR-014**: repair 後、統合により特徴が変化するレース(2025 後半以降/2026)の予測を再生成できる MUST。
- **FR-015**: repair 後、materialize 済み特徴(存在する場合)を再生成できる MUST。

**不変条件・境界(全 US)**

- **FR-016**: identity 解決・repair はいずれもモデルの特徴計算ロジックを変更してはならない MUST NOT(これはデータ修復であり特徴変更ではない)。採用済みモデルの serving 互換(特徴バージョン/ハッシュ)を維持 MUST。
- **FR-017**: 統合前後で、統合対象のサロゲートが関与しない過去レース(JRA-VAN 期・strictly-before で netkeiba 走が集約に入らないレース)の特徴は**変化してはならない** MUST NOT(parity)。
- **FR-018**: `id_mappings` およびその解決結果は、モデルの特徴として流入してはならない MUST NOT(リーク境界)。
- **FR-019**: 本 feature はスキーマ変更を伴わない MUST(既存の `id_mappings` 列を用い、migration を追加しない)。

### Key Entities

- **ID マッピング**: netkeiba の `source_id` と canonical ID の対応。状態(未マッピング/解決済み/要確認)と解決根拠・時刻を保持する。本 feature の中心的な操作対象。
- **馬 / 騎手 / 調教師 マスタ**: canonical ID を主キーに持つ個体。サロゲートと canonical に分裂しうる。統合の帰着先。
- **出走 / 結果 / 予測 / 特徴スナップショット**: 個体IDを保持する明細。repair の寄せ替え対象。
- **推奨(買い目)**: 選択内容に馬IDを内包する明細。寄せ替えでなく再生成で整合。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: identity 解決後、規則で自動 `mapped` 化されるサロゲート(馬 ≈5,975・騎手 ≈152・調教師 ≈198)が repair で canonical へ統合され、対応する分裂が 0 になる。番号一致だが照合不成立の残り(馬 2・騎手 ≈11・調教師 ≈9)は自動統合せず conflict として分離され、手動解決後に別途統合される。
- **SC-002**: 統合された馬について、直近(2026)レースの予測特徴が実キャリアの過去走を反映する(サンプル馬で「デビュー扱い(過去走 0)」が解消される)。
- **SC-003**: 統合対象が関与しない過去レース(JRA-VAN 期)の特徴が、統合前後でバイト単位一致する(parity)。
- **SC-004**: repair を 2 回連続実行しても、2 回目は追加変更 0 件(冪等)。
- **SC-005**: 番号一致だが属性不一致のケース(馬 2 頭)および番号不一致の真の未マッピング(サロゲートの残り 38%)が、自動統合されずに保留/未マッピングのまま維持される(誤統合 0)。
- **SC-006**: 取り込み時 identity チェック導入後、既存 canonical ID と一致する個体を取り込んでも新規サロゲートが作成されない(新規分裂 0)。
- **SC-007**: 馬・騎手・調教師の詳細ページで、統合された個体の全出走履歴が単一IDで一体表示される(サヴォーナで 19 走)。

## Assumptions

- 番号一致(`source_id == 既存 canonical ID`)に加え名前一致を伴えば、同一個体である蓋然性は極めて高い(実測で馬 5,977/5,977 名前一致・名前不一致 0)。これは「名前だけ」「番号だけ」の推測結合とは区別され、identity(構造的同一)+ 裏取りとして扱う。
- netkeiba 由来レースは同一レース内に JRA-VAN 個体を混在させない(実測: netkeiba 個体が出走する 2,658 レースすべてが純 netkeiba)。したがって寄せ替え時のレース内衝突は構造的に発生しない(実測 PK 衝突 0 件)。それでも衝突ガードは防御的に必須。
- as-of 特徴は strictly-before で集約されるため、統合で追加される netkeiba 走(2026=後方)は過去レース(pre-2025)の集約に入らない。ゆえに過去特徴は不変で、変化するのは統合走を過去に持つ直近レースのみ。
- 現行の採用済みモデルの学習窓は pre-2025 に収まるため、本 feature 単体での再学習は不要。将来の再学習が正しい統合履歴の恩恵を受ける。
- repair は operator 実行の冪等 CLI(feature materialize / live refresh / predict-backfill と同型)として提供され、schema migration を伴わない。
- 物理寄せ替え(re-key)は仮想統合(全読み経路で `id_mappings` を辿る)より下流コード改修面が小さく保守的、という判断に基づく。仮想統合は不採用。

## Honest Limitations

- 第三者(codex)設計レビュー取得済み(P0 指摘 8 件を全反映)。ただし識別根拠は実 DB 実測と静的レビューであり、実装後の実 DB E2E で最終確認する。
- 物理 re-key は非可逆寄り(`id_mappings`/取込ジョブに監査記録は残るが、元サロゲートのマスタ行は消える)。ゆえに dry-run による事前確認と、repair 中の writer 停止(または排他ロック)を必須とする。
- 騎手/調教師の prefix 照合は番号名前空間の意味的同一に依存する(免許再利用・外国人短期免許・同姓 prefix の別人リスク)。初回 backfill は dry-run 候補一覧の operator 承認をゲートとする。conflict(馬 ≈2・騎手 ≈11・調教師 ≈9)の手動レビュー運用は最小限に留め、本格的な運用フロー/UI は範囲外。

## Out of Scope (Deferred)

- conflict の本格的な運用 UI/レビューフロー。
- 番号不一致の真の未マッピング(残り 38%)を別ソース照合等で解決すること。
- JRA-VAN フィード復旧時の再統合ロジック。
- 表示のみの先行対処(案A)。repair 後は分裂が自動的に解消されるため、旧サロゲートURLの誘導/未収録応答以外は不要。
