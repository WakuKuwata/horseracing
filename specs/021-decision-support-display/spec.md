# Feature Specification: 意思決定支援の表示強化 (Decision-Support Display)

**Feature Branch**: `021-decision-support-display`

**Created**: 2026-06-28

**Status**: Draft

**Input**: User description: モデル p と市場 q の併記・予測校正の可視化・予測の不確実性/信頼度表示。製品の目的は「市場を自動で超えて確実に勝つ」ことではなく、人間が予測・確率・期待値とその信頼性を見て自分で購入判断するための『正直で正確な意思決定支援』。

## 背景 (Why)

Feature 020 の実データ評価で、公開情報ベースの特徴をいくら足してもモデル p は市場 q を超えられないことが確認された（市場 q の win LogLoss 0.202 < モデル p 0.234、p−q edge は実現勝率と逆相関）。しかし本製品は**人間の意思決定支援ツール**であり、市場に勝てない状態でも「正確で正直な予測・確率・期待値・その信頼性」を提示すること自体に価値がある。市場 q は敵ではなく、人間に見せる比較材料である。本 feature は既存の read-only 意思決定支援（014 API + 015 front）を、その価値が伝わる表示へ強化する。**市場超過・自動収益は目的にしない。**

## User Scenarios & Testing *(mandatory)*

### User Story 1 - モデル p と市場 q の併記表示 (Priority: P1)

レース詳細を見る人が、各馬について「モデルの予測勝率 p」と「市場が考える勝率 q」を並べて確認し、両者の乖離（どちらが強気か）を一目で把握して、自分の購入判断に使える。

**Why this priority**: 市場 q はモデルより正確な強力な比較基準であり、p と q を並べて見せること自体が最も直接的な意思決定支援になる。MVP として単独で価値を提供する。

**Independent Test**: あるレース詳細を開き、各出走馬に対しモデル p と市場 q が別々の値として並び、差分（p−q）と「どちらが強気か」が表示され、q が「市場推定（vote-share、真の確率でもモデルでもない）」と明示ラベルされていることを確認できる。

**Acceptance Scenarios**:

1. **Given** 有効な win オッズと予測がある出走馬, **When** レース詳細を開く, **Then** その馬のモデル p と市場 q が別フィールドで並び、p−q の乖離が分かる形で表示される。
2. **Given** win オッズが欠損した馬, **When** レース詳細を開く, **Then** その馬の q は 0 ではなく「未提供（--）」として安全に表示される（p は表示）。
3. **Given** 市場 q が表示される, **When** 画面を見る, **Then** q には「市場推定・favorite-longshot bias を含む・真の確率でもモデル p でもない」旨のラベルが必ず付く。

---

### User Story 2 - 予測校正の可視化 (Priority: P2)

人間が「モデルが p=30% と言ったら実際に約 30% 勝つのか」を確認できる校正図（reliability 曲線）を見て、表示される確率をどれだけ信頼してよいか判断できる。

**Why this priority**: 確率の信頼性が分かって初めて、p や EV を意思決定に使える。校正の可視化は「数字の信頼度」を担保する。

**Independent Test**: 指定したモデルバージョン・期間について、予測確率帯ごとの「予測平均 vs 実現勝率 + サンプル数 + 全体 ECE」が取得・表示され、retrospective（過去レース結果に基づく）であることとサンプル数・期間・model_version が監査表示されることを確認できる。

**Acceptance Scenarios**:

1. **Given** 過去予測と結果が永続化されている, **When** 校正ビューを開く, **Then** 予測確率帯ごとの予測平均と実現勝率が対比表示され、全体 ECE とサンプル数が出る。
2. **Given** 校正ビューを見る, **When** 内容を確認, **Then** 「過去レース結果に基づく retrospective 表示・採用ゲートではない」旨と model_version/期間が明示される。

---

### User Story 3 - データ裏付け（条件カバレッジ）の表示 (Priority: P3)

人間が個々の予測について「この確率はどれくらいデータに裏付けられているか」の目安を見て、確率値だけを過信しないで判断できる。**汎用的な「信頼度/正確さの保証」ではなく、検証可能でリーク安全な「データ裏付け（条件カバレッジ）」に限定する**（codex R6/US3 verdict 反映）。指標は事前レース情報のみ（例: 出走頭数、その条件の学習サンプル数、学習データの時代的近さ）に基づき、結果・オッズ・表示派生値を一切使わない。

**Why this priority**: 同じ p=20% でも、学習データが薄い条件では裏付けが弱い。確率の過信を防ぐ補助情報。ただし「予測が当たる確信」を意味すると誤解させるリスクが高いため、意味を「データ裏付け」に厳密に限定し、具体定義は計画段階で確定・過去データで妥当性確認する（妥当な指標が定まらなければ本 US は defer 可）。

**Independent Test**: 各予測にリーク安全な「データ裏付け」の目安（高/低など、粗い注意バッジ）が付与され、裏付けの弱いケースが視覚的に区別でき、その指標が結果・オッズ・表示派生値を参照していない（事前レース情報のみ）ことを確認できる。

**Acceptance Scenarios**:

1. **Given** 予測が表示される, **When** レース詳細を見る, **Then** 各確率に「データ裏付け」の目安が併記され、裏付けの弱いケースが区別でき、「予測の的中確信ではない」旨が明示される。
2. **Given** データ裏付け指標, **When** その算出根拠を確認, **Then** 対象レースより前の情報のみを用い、結果・オッズ・表示派生値を参照していない。

---

### Edge Cases

- 全馬の win オッズが欠損 → q は全馬「未提供」、p のみ表示（落ちない）。
- スクラッチ馬 → 確率は canonical field（除外＋再正規化）に従い、p/q とも矛盾なく扱う。
- 校正対象期間にサンプルがほぼ無い → 「サンプル不足」を明示し、誤った reliability を断定表示しない。
- 予測 run が無い / 結果未確定のレース → 校正は対象外、p/q 併記は予測がある範囲で表示。
- pseudo/推定値（q・推定オッズ・pseudo-ROI）→ 必ずラベル経由で表示、実現値と混同させない。

## Requirements *(mandatory)*

### Functional Requirements

**US1 — p と q の併記**
- **FR-001**: API は出走馬ごとに、モデル win 確率 p と市場推定 win 確率 q（= (1/オッズ)/Σ(1/オッズ)、有効オッズを持つ出走馬で正規化）を**別フィールド**の read-only データとして公開しなければならない（混同禁止）。
- **FR-001a**（codex R1/R10）: p と q は**同一の canonical field（スクラッチ除外＋再正規化した母集団）**上で算出しなければならない。p と q の母集団が揃わない場合、乖離（p−q）は数学的に無意味なので front は乖離表示を抑制し「データ品質上 比較不可」状態を示さなければならない（部分母集団で乖離を計算しない）。
- **FR-002**: q は「市場推定（vote-share）・favorite-longshot bias を含む・真の確率でもモデル p でもない」と明示ラベルされなければならない。
- **FR-002a**（codex R4）: 生の市場 q と FL 補正後 q'（013）は別の問いに答える別フィールドであり、同じ `q` ラベルを使い回してはならない。市場比較には生 q を用い、q' を表示する場合は独立フィールド＋ラベルとする。
- **FR-003**: front は各馬の p と q を併記し、乖離（p−q）を単一の一貫した描画経路で表示しなければならない。
- **FR-003a**（codex R3, 最重要）: 乖離は「**モデルと市場の意見の相違**」として中立に提示し、**「買い/お買い得」等の利益示唆の言葉・緑赤の損益色・p−q によるソートやハイライトを用いてはならない**（p−q>0 を買いシグナルと短絡させない。020 で市場 q の方が予測上手いと実証済み）。
- **FR-004**（codex R10）: q は win オッズのみから算出し、有効オッズが無い馬は q を未定義とし 0 で補完せず安全表示（--/未提供）にしなければならない。各馬で as_of・オッズ source・スクラッチ状態を表示する。
- **FR-005**: p（モデル）と q（市場）は端から端まで別物として保持し（p≠q）、表示する p/q いずれの値もモデルの学習特徴に戻してはならない。

**US2 — 校正の可視化**
- **FR-006**: API は指定 model_version・期間について、予測確率帯ごとの「予測平均・実現勝率・**件数**」と全体 ECE を read-only で公開しなければならない。
- **FR-006a**（codex R2, 最重要）: 校正は既定で **walk-forward OOS の保存済み予測**で算出しなければならない。in-sample / 事後再計算値は実予測時点より良く見えるため、再計算を用いる場合は "recomputed diagnostic" として明示ラベル（PseudoBadge 相当）＋出典を付さなければならない。
- **FR-006b**（codex R5）: bin の件数を前面に表示し、少数件 bin は統合または「件数不足」として抑制しなければならない。ECE は記述的診断として扱い、統計的不確実性（信頼区間や「有意でない」旨）を併記する。
- **FR-007**（codex R8）: 校正のスコープは監査可能な **model_version + prediction_run** に明示限定し、カバレッジが不完全な場合は黙ってバックフィルせず「カバレッジ範囲外」を明示しなければならない。model_version・期間・サンプル数・予測出典を監査情報として伴う。
- **FR-008**: front は予測確率 vs 実現勝率の reliability 曲線/図を bin 件数付きで表示し、人間が「p の信頼性」を判断できるようにしなければならない。
- **FR-009**: 校正表示は情報提供であって採用ゲートではなく、いかなる書き込み・モデルへの影響も行ってはならない。

**US3 — データ裏付け（条件カバレッジ）**
- **FR-010**（codex R6）: API は予測ごとにリーク安全な「データ裏付け（条件カバレッジ）」の目安を公開しなければならない。指標は事前レース情報のみ（出走頭数・その条件の学習サンプル数・学習データの時代的近さ 等）に基づき、結果・オッズ・表示派生値を混入させてはならない。
- **FR-011**: front は確率に「データ裏付け」の目安を併記し、裏付けの弱いケースを視覚的に区別しつつ、「予測の的中確信ではない」旨を明示しなければならない（過信防止）。粗い注意バッジ以上の意味を持たせない。
- **FR-012**: 「データ裏付け」指標の具体定義は計画段階で確定し、リーク安全であり過去データで妥当性を確認しなければならない。妥当な指標が定まらない場合、本 US は defer 可とする。

**横断（憲法整合 + codex 抜け表示）**
- **FR-013**（codex R7）: 新しい表示データはすべて read-only の API で公開し（014 は read-only 厳守・新規書き込み経路を作らない）、OpenAPI 記述・nullability・source ラベル・監査メタ・警告セマンティクスを UI 実装前に契約として確定し、front 型は committed openapi から自動生成し drift-check を維持しなければならない（憲法 VI）。
- **FR-014**: pseudo/推定値（q・q'・推定オッズ・pseudo-ROI・recomputed 校正 等）は単一の PseudoBadge/SourceBadge 経路でラベル表示し、実現値と誤認させてはならない（憲法 V）。
- **FR-015**: audit（model_version/logic_version/computed_at/as_of、オッズの取得時刻・source＝確定/事前推定）を画面に表示しなければならない。
- **FR-016**: 確率は 009 の整合性（win→joint）を維持しなければならない。スキーマ変更は原則行わず（reliability/q は永続化済みデータから算出）、必要時のみ計画で justify する。
- **FR-017**（codex 抜け表示）: 「市場 q の方がモデル p より予測が上手い（020 実証）」事実を画面で明示し、ユーザーがモデルを過信しないようにしなければならない。
- **FR-018**（条件付き・将来制約／analyze G1）: 021 は EV 表示を新規追加しない（スコープ外、契約に EV フィールドを足さない）。**将来** EV を表示する際は、控除率（takeout）とその出典、および「EV は p の誤差に敏感（O−1 小で敏感）」を必ず併記する（本 feature では適用対象なし）。
- **FR-019**（codex R9）: API/表示出力は一方向の read-only 意思決定支援成果物であり、q/q'/p−q/reliability/EV/データ裏付け のいずれの表示派生値もモデル学習特徴に流用してはならない（leak-guard テストで担保）。

### Key Entities *(include if feature involves data)*

- **市場推定 win 確率 q**: 出走馬ごとに win オッズから導出する推定確率（vote-share）。真の確率でもモデル p でもない。
- **モデル win 確率 p**: active な予測 run からの各馬の予測勝率。q とは別物。
- **校正サマリ (reliability)**: model_version＋期間ごとの、予測確率帯 → 実現勝率・件数、全体 ECE。過去結果に基づく retrospective。
- **予測信頼度指標**: 予測ごとのリーク安全な信頼度/不確実性の目安（定義は計画で確定）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 任意のレース詳細で、全出走馬についてモデル win % と市場推定 win % の両方と差分が 1 画面で確認できる。
- **SC-002**: 市場推定値・pseudo 値は常にラベル付きで表示され、UI 監査で「ラベルなしの pseudo/推定値が 0 件」である。
- **SC-003**: 指定モデル・期間について、確率帯ごとの予測 vs 実現勝率の校正図がサンプル数付きで閲覧でき、retrospective である旨が明示される。
- **SC-004**: 表示される各確率に信頼度の目安が付き、低データ（低信頼）ケースが高信頼ケースと視覚的に区別できる。
- **SC-005**: 新しい表示データは 100% read-only で提供され、新規書き込みエンドポイントが導入されず、front 型が committed API 契約と一致する（drift-check 緑）。
- **SC-006**: 表示するいずれの値（p/q/q'/p−q/EV/校正/データ裏付け）もモデル特徴に逆流しない（leak-guard で検証）。
- **SC-007**（codex R3）: p−q 表示に利益示唆の言葉・損益色・edge ソート/ハイライトが 0 件であり、「モデルと市場の意見相違」として中立提示される。
- **SC-008**（codex R1/R2）: 乖離表示は p と q が同一 canonical field のときのみ行われ（不一致時は比較不可表示）、校正は既定で walk-forward OOS 由来であることが画面で判別できる。
- **SC-009**（codex 抜け表示）: 「市場の方が予測上手い」事実・オッズの as_of/source が画面で確認できる（EV/控除率の表示は 021 スコープ外、FR-018 参照）。

## Assumptions

- 市場 q は API が公開する win オッズから導出する（FL-bias 補正 013 の表示併用は計画で要否判断）。
- 校正は eval の既存 ECE/reliability 計算を、永続化済みの過去予測＋結果に対して用いる（独自の新指標を勝手に作らない）。
- 不確実性指標の具体定義（例: 出走頭数・データ十分さ・条件の希少さ等に基づく信頼度）は計画段階で確定する。リーク安全を必須とする。
- 本 feature は 014（API）/015（front）を拡張する。014 は read-only を維持。必要なら eval の校正出力を read。
- スキーマ変更は原則なし。日本語 UI 規約を維持。

## codex レビュー所見 (top risks folded)

新規 spec として codex second opinion を実施（CLAUDE.md 方針）。反映した主なリスク:
- **R1/R10 (Critical)**: p と q は同一 canonical field で算出。母集団不一致なら乖離表示を抑制、q をゼロ補完しない → FR-001a/FR-004。
- **R2 (Critical)**: reliability は既定で walk-forward OOS 保存予測。再計算は診断ラベル → FR-006a。
- **R3 (High)**: p−q を買いシグナルと短絡させない（profit 言語/損益色/edge ソート禁止、「意見相違」提示）→ FR-003a/SC-007。
- **R4 (High)**: 生 q と FL 補正 q' は別フィールド・別ラベル → FR-002a。
- **R5 (High)**: bin 件数を表示、少数 bin 抑制、ECE は不確実性付き診断 → FR-006b。
- **R6/US3 verdict**: 漠然とした「信頼度」を「データ裏付け（条件カバレッジ）」に限定（事前情報のみ・検証必須・不定なら defer 可）→ US3 全面改稿/FR-010〜012。
- **R7 (High)**: API 契約（nullability/source/監査/警告セマンティクス）を UI 前に確定、drift-check → FR-013。
- **R8 (Medium)**: 校正 scope を model_version+prediction_run に限定、カバレッジ欠落を明示 → FR-007。
- **R9 (Medium)**: 表示派生値はモデル特徴に流用しない（leak-guard）→ FR-019/SC-006。
- **抜け表示**: 020 の「市場優位」事実・控除率と EV 前提・オッズ as_of/source の明示 → FR-017/018/015、SC-009。

## Scope (Out / Deferred)

- real exotic odds の本番取得（012 の運用、別タスク）
- 市場超過・auto-betting・実ステーク運用
- **EV/控除率/Kelly 等の金額・期待値表示**（021 は p/q・校正・データ裏付けの表示に限定。将来 EV を出す際に FR-018 を適用）
- 生 q と FL 補正 q'(013) の併記（021 は生 q のみ。q' を出す場合は別フィールド・別ラベル＝FR-002a を将来適用）
- 新シグナル（sectional/ラップ/脚質 等）の取得
- auth、prod デプロイ、Playwright E2E
