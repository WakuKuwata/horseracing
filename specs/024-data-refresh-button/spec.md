# Feature Specification: netkeiba データ更新ボタン

**Feature Branch**: `024-data-refresh-button`

**Created**: 2026-06-28

**Status**: Draft

**Input**: User description: "レース一覧ページとレース詳細ページに『データ更新』ボタンを置き、押すと netkeiba から最新データを取得して DB に反映する。詳細=1レース、一覧=その日の全レース。非同期ジョブ＋進捗表示。014 read-only API には write を足さず別 ops サービスを新設。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 詳細ページで 1 レースを更新する (Priority: P1)

レース詳細を見ているユーザーが、表示中の 1 レースについて netkeiba の最新状態（出馬表・オッズ、あるいは確定後なら結果）を取り込みたい。詳細ページの「このレースを更新」ボタンを押すと、取得がバックグラウンドで始まり、ユーザーは画面を離れずに進捗（受付中 → 取得中 → 完了/一部失敗/失敗）を確認でき、完了すると最新データで表示が更新される。

**Why this priority**: 最小で価値が出るスライス。1 レース単位は負荷が軽く、失敗時の影響範囲も小さいため、更新フローの土台（受付・進捗・完了後の再表示・失敗時の扱い）を最初に確立できる。これ単体で「ボタンで最新化できる」という製品価値が成立する。

**Independent Test**: 任意の 1 レースの詳細ページでボタンを押し、(a) 即座に受付応答が返る、(b) 進捗が更新中状態を経て終了状態に到達する、(c) 取得が成功すると詳細の表示内容（出走馬・オッズ・結果のいずれか）が更新後の値に変わる、ことを確認すれば独立して検証できる。

**Acceptance Scenarios**:

1. **Given** 結果待ち（結果未確定）のレース詳細を開いている, **When** 「このレースを更新」を押す, **Then** 即座に「受付済み」状態になり、出馬表とオッズの取得がバックグラウンドで進み、完了後に表示が最新化される。
2. **Given** 結果確定後のレース詳細を開いている, **When** 「このレースを更新」を押す, **Then** 結果データが取得され、確定済み結果が表示に反映される（既存の確定データは保護され、上書きで壊れない）。
3. **Given** 取得が一部のデータでしか成功しなかった, **When** ジョブが終了する, **Then** 「一部成功」であることがユーザーに分かる形で示され、何が取れ何が取れなかったかが区別できる。
4. **Given** 取得が失敗した（ネットワーク不通・対象ページ無し等）, **When** ジョブが終了する, **Then** 失敗状態が明示され、既存の表示データは壊れず保持される。

---

### User Story 2 - 一覧ページでその日の全レースを更新する (Priority: P2)

特定開催日のレース一覧を見ているユーザーが、その日の全レースをまとめて最新化したい。「この日を更新」ボタンを押すと、日付内の各レースの取得がバックグラウンドで進み、ユーザーはレース単位の進捗（どれが完了・取得中・失敗か）を一覧上で把握でき、失敗したレースだけを後から再実行できる。

**Why this priority**: 実運用での利便性が高いが、複数レースを同時に扱うため並列度の制御・重複起動の抑制・部分失敗の扱いが必要で、US1 で土台を固めた後に乗せるのが妥当。US1 が無くても価値はあるが、規模が大きく失敗時の影響も広い。

**Independent Test**: ある開催日の一覧ページで「この日を更新」を押し、(a) その日の各レースに対して取得が起動する、(b) レースごとの進捗・終了状態が一覧上で区別して見える、(c) 一部が失敗した場合に失敗分だけを再実行できる、ことを確認すれば独立して検証できる。

**Acceptance Scenarios**:

1. **Given** 12 レース×複数開催のある日付の一覧を開いている, **When** 「この日を更新」を押す, **Then** その日の全レースに対して取得が受け付けられ、レースごとの進捗が表示される。
2. **Given** 一括更新が進行中, **When** 一部のレースだけ取得が失敗する, **Then** 成功・失敗がレース単位で区別され、失敗したレースだけを再実行する手段が提供される。
3. **Given** その日の一括更新が直前に走り終えた直後, **When** 同じ「この日を更新」をすぐ再度押す, **Then** netkeiba への過剰アクセスを避けるため、まだ有効な直近結果が再利用される／重複起動が抑制される。

---

### User Story 3 - 重複起動の抑制と更新履歴の可視化 (Priority: P3)

更新を何度も押す・複数ユーザーが同時に押すといった状況でも、同一レースの取得が多重に走って netkeiba を叩きすぎないようにし、いつ・何を・どの結果で更新したかの履歴を後から確認できるようにする。

**Why this priority**: 運用上の安全弁と監査。US1/US2 が機能した上で、礼儀正しさ（rate-limit 尊重）と監査性を仕上げる位置づけ。無くても基本機能は動くが、無いと netkeiba への過負荷や「いつ更新したか分からない」問題が残る。

**Independent Test**: 同一レースに対して短時間に複数回更新を要求し、(a) 取得が多重に走らない（既存の進行中ジョブが再利用される）、(b) 各更新の実行記録（対象・種別・結果・件数・時刻）が残り後から参照できる、ことを確認すれば独立して検証できる。

**Acceptance Scenarios**:

1. **Given** あるレースの更新ジョブがまだ実行中, **When** 同じレースの更新をもう一度要求する, **Then** 新しい取得は起動されず、進行中のジョブの状況が返る。
2. **Given** あるレースの更新がごく最近成功して結果が新鮮, **When** 同じレースの更新を要求する, **Then** 不要な再取得を避け、直近結果の再利用が選ばれる（強制再取得の明示要求がある場合を除く）。
3. **Given** 複数回の更新が行われた, **When** 履歴を参照する, **Then** 各実行の対象レース・取得種別・結果状態・処理件数・実行時刻が記録されている。

---

### Edge Cases

- **不正なレース識別子**: 12 桁の正規 ID でない、または存在しないレースに対する更新要求は、取得を起動せず明確なエラーで拒否される。
- **netkeiba 側にデータがまだ無い**: 出馬表が未公開のレースを更新した場合、取得は失敗または「データ無し」として終了し、既存表示は壊れない。
- **確定データの保護**: 確定済み結果（信頼できる元データ由来）が、後発の取得で誤って上書き・破壊されない。
- **結果待ちレースのオッズ更新**: 結果未確定レースの事前オッズは最新値で上書きしてよいが、確定後のオッズ／結果は保護対象として扱う。
- **取得途中での状態変化**: 更新要求の受付後にレースが「結果待ち→確定」へ変わった場合でも、取得種別の判定が破綻せず、不整合な書き込みをしない。
- **大量同時起動**: 一覧の一括更新で多数のレースが同時要求されても、netkeiba への同時アクセスが上限内に保たれる。
- **進捗の見失い**: ページ再読込・離脱後の再訪でも、進行中／直近の更新状況を再取得して提示できる。

## Requirements *(mandatory)*

### Functional Requirements

**更新の起動とスコープ**

- **FR-001**: ユーザーはレース詳細ページから、表示中の 1 レースのデータ更新を要求できる（US1）。
- **FR-002**: ユーザーはレース一覧ページから、開催日の全レースのデータ更新を一括で要求できる（US2）。**(改訂 2026-06-28, 追補A)** 対象レースは **netkeiba の開催日ページから発見**する（DB に未登録の日でも可）。発見は worker 側で行い、要求は即 202 を返す（FR-003 と整合、リクエスト経路で netkeiba を叩かない）。
- **FR-003**: システムは更新要求を受けると即座に「受付」を返し、ユーザーを待たせずにバックグラウンドで取得を進めなければならない（同期で完了を待たせない）。
- **FR-004**: システムは更新要求ごとに追跡用の識別子を払い出し、ユーザー（フロント）はそれを使って進捗・終了状態を後から問い合わせできなければならない。

**取得内容の出し分け**

- **FR-005 / FR-006 (改訂 2026-06-28, 追補A)**: 1 レースの更新は **entries + results + odds の全パスを 1 回で実行**する（既存の安全規則で状態に依らず安全: results は INSERT-only、odds は確定後 fill-null・未確定は上書き）。当初の「未確定→entries+odds / 確定→results」分岐は、**未取り込みの確定済みレースが「結果行が無い＝未確定」と誤判定され結果を取りこぼす**ため廃止。未公開（未来）レースは結果ページが無く results サブ手順だけ benign に失敗し、ジョブは FAILED でなく PARTIAL になる。
- **FR-007**: 取得したデータは既存のコアデータ表示（出走馬・オッズ・結果）に反映され、ユーザーは更新後に最新値を閲覧できる。

**進捗・結果のフィードバック**

- **FR-008**: システムは各更新について「受付済み／取得中／成功／一部成功／失敗／対象なし（取得対象が未公開・無しでスキップ）」を区別できる状態を提供しなければならない。
- **FR-009**: 一括更新（US2）では、進捗・終了状態をレース単位で区別して提示できなければならない。
- **FR-010**: 一括更新で一部のレースが失敗した場合、ユーザーは失敗したレースだけを再実行できなければならない。
- **FR-011**: フロントは取得完了を検知したら、最新データを取得し直して表示を更新しなければならない。
- **FR-012**: 取得が失敗・一部失敗でも、既存の表示データは破壊されず保持されなければならない。
- **FR-013**: フロントは「読み込み中／空（正常だが0件）／型付きエラー」を区別した UI 状態を保ち、更新の進行中・失敗を既存の表示原則に沿って提示しなければならない。

**礼儀正しさ・重複抑制**

- **FR-014**: システムは netkeiba への取得において、既存の取得基盤の礼儀正しさ（アクセス間隔・リトライ抑制・robots 尊重）を維持しなければならない。
- **FR-015**: 同一レースについて取得が進行中、または直近に成功して結果が十分新鮮な場合、システムは重複した取得を起動せず、進行中ジョブの状況または直近結果の再利用を返さなければならない（明示的な強制再取得要求がある場合を除く）。
- **FR-016**: 一括更新における netkeiba への同時アクセス数は上限内に制御されなければならない。

**監査**

- **FR-017**: システムは各更新の実行記録（対象レース・取得種別・結果状態・処理件数・実行時刻）を残し、後から参照できなければならない（既存の取り込み監査の仕組みを流用）。

**データ保護・境界（憲法）**

- **FR-018**: 確定済み結果は後発の取得で上書き・破壊されてはならない（追記的に扱う）。事前オッズの上書きは結果未確定レースに限る。
- **FR-019**: 正規の 12 桁レース識別子でない、または存在しないレースへの更新要求は、取得を起動せず拒否されなければならない。
- **FR-020**: 取り込んだオッズ・結果は予測モデルの特徴量に流入してはならない（リーク防止、憲法 II）。
- **FR-021**: 既存の読み取り専用データ閲覧（予測・オッズ・推奨の表示）は、本機能の追加によって書き込み可能になってはならない。更新（書き込み）の経路は、読み取り専用の閲覧経路から分離されていなければならない（憲法 VI）。
- **FR-022**: 推定値・疑似値（推定オッズ・疑似 ROI 等）は、更新後の表示でも引き続きラベル表示され、実績と誤読されてはならない（憲法 V）。

### Key Entities *(include if feature involves data)*

- **更新ジョブ (Refresh Job)**: 1 回の取得作業を表す。対象レース、取得種別（出馬表＋オッズ／結果）、状態（受付済み・取得中・成功・一部成功・失敗）、処理件数・エラー件数、開始・完了時刻、結果サマリを持つ。既存の取り込み監査エンティティを流用する。
- **一括更新 (Day Refresh Batch)**: ある開催日に対する一括更新。配下に複数の更新ジョブ（レース単位）を束ね、全体の進捗とレース別の状態を提供する。
- **レース (Race)**: 既存エンティティ。正規の 12 桁識別子を持ち、結果確定の有無により取得種別が決まる。
- **コアデータ (出走馬・オッズ・結果)**: 既存エンティティ。更新の書き込み先であり、確定結果は保護対象。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーがボタンを押してから「受付済み」が画面に反映されるまでが体感的に即時（待ち時間で取得完了をブロックされない）。
- **SC-002**: 詳細ページの 1 レース更新で、取得が成功した場合に表示中のデータ（出走馬・オッズ・結果のいずれか）が更新後の値へ変わることを、ユーザーが操作だけで確認できる。
- **SC-003**: 一括更新でその日の全レースについて、レース単位の成功／失敗が一覧上で区別して見え、失敗分だけを再実行できる。
- **SC-004**: 同一レースへ短時間に複数回更新を要求しても、netkeiba への取得が多重に走らない（重複起動が抑制される）。
- **SC-005**: 更新が失敗・一部失敗した場合でも、既存の表示データが壊れない（更新前に見えていた確定情報が消えない）。
- **SC-006**: 読み取り専用の閲覧経路に書き込み手段が一切増えていない（閲覧 API は引き続き参照のみ）。
- **SC-007**: 全ての更新について、対象・種別・結果・件数・時刻の記録が後から参照できる。

## Assumptions

- **netkeiba を唯一の取得元**とする（複数ソース統合は対象外）。取得は既存のスクレイピング基盤（実パーサ・礼儀正しいフェッチ）を流用する。
- レース識別子（12 桁 JRA-VAN ID）から netkeiba の取得 URL を導出できる既存手段を用いる。それ以外の高度な ID 解決は対象外。
- 更新（書き込み）は、読み取り専用の閲覧サービスとは別の経路で行う。閲覧サービスは参照専用のまま変更しない。
- 取得結果の DB 反映ルール（確定結果の保護＝追記のみ、事前オッズは結果未確定時のみ上書き、識別子マッピング）は既存の取り込みルールを踏襲する。
- 「直近に成功して新鮮」の判定に用いる鮮度のしきい値は実装時に妥当な既定値を定める（運用で調整可能）。
- 本機能は人手のボタン操作によるオンデマンド更新が対象。定期実行（スケジューリング／cron）・自動運用は対象外。
- 認証／認可、リアルマネー連携、Kelly 等の数値ロジック変更は対象外（deferred）。
- ジョブの実行・進捗管理は既存の取り込み監査エンティティ（ジョブ状態を持つ仕組み）を土台に構成できることを前提とする。

## 追補A (2026-06-28): 開催日ディスカバリ + 500 修正

**背景**: `POST /ops/v1/days/{date}/refresh` が 2025-10-12 で 500 になった。原因は (1) 当初の `enqueue_day` が DB の `races` 行のみを対象とし netkeiba から発見していなかった、かつ (2) レース 0 件の 404 を返す**前に**親ジョブを INSERT(flush) しており、書き込み不可ロール（018 の app_ro 等）だと flush が permission denied → 500（owner ロールなら 404 + 孤児ジョブ）。

**変更**:
- **発見は worker 側**。`refresh_day` の POST は親ジョブ(QUEUED)を 1 件作って即 202 を返すだけ（netkeiba を叩かない＝FR-003）。worker が `refresh_day` を claim → `discover_races(date)`（US4, netkeiba 開催日フラグメント）で当日全 race_id を発見 → 各 race に `refresh_race` 子を fan-out（親の trace_id 共有）。0 件なら親 SUCCEEDED の clean no-op。
- **404 廃止**: 任意の日付を受理（未取り込みの日も worker が発見）。空書き込み（404前の親 INSERT）を除去し、書き込みは正規の enqueue のみ＝500 の経路を解消。
- **batch ポーリング**: 発見前（子 0 件）でも親を認識して 200 を返す（`/batches/{trace_id}` が 404 にならない）。状態は子があれば集約、無ければ親の状態（queued/running）を反映。
- **runner 全パス化**: FR-005/006 参照（entries+results+odds を 1 回、安全規則維持）。
- **取得層**: `make_fetcher` に httpx クライアントを付与（従来クライアント無しでキャッシュ済みページしか取得できず**ライブ取得不能**だった潜在バグを修正）。

**既知の軽微な edge**: 同一日に 2 回 POST すると、2 回目の親は `enqueue_race` の重複排除で既存(1 回目)の子を再利用するため**子 0 件の batch**になる（実取得は重複しない＝意図どおりだが、2 回目の batch 表示は空）。単一クリック運用では問題なし。

**実データ検証**: `POST /ops/v1/days/2025-10-12/refresh` → **202**（旧 500）、worker が **24 レースを発見**し fan-out、子を drain して entries+results+odds を取得（実 netkeiba）。30 ops テスト緑。
