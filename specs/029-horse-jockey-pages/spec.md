# Feature Specification: 馬・騎手プロフィールページ

**Feature Branch**: `029-horse-jockey-pages`

**Created**: 2026-06-29

**Status**: Draft

**Input**: User description: "馬のデータ画面を作成して表示する。基本的な horse のデータ表示／血統情報の表示／特徴量や parquet などのデータも表示。同様に騎手情報も表示するページ。遷移ルートはレースの詳細画面から。"

## Clarifications (確定済みの設計判断)

- **特徴量・parquet 表示は MVP では出さない**（後続 feature に defer）。理由: 014 read-only API は `db + probability` のみ依存で、parquet/features を読むと api→features/pandas 結合が発生し read-only・最小依存原則と衝突。特徴量は「対象レース基準の as-of 値」で文脈が要り、生の特徴量ダンプは開発者向けの色が濃く製品目的（正直な意思決定支援）とずれる。→ 本 feature は**事実データ（DB 集計）のみ**。
- **騎手ページは馬ページと同時リリース**（US1 馬 + US2 騎手）。
- 血統は ID がほぼ未投入（実 DB で sire_id 等 ~0%）のため**名前で表示**（父／母／母父の名前）。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 馬プロフィールを見る (Priority: P1)

レース詳細で出走馬を見ているユーザーが、ある馬の素性をもっと知りたい。馬名をクリックすると馬プロフィールページに遷移し、その馬の基本情報（名前・性・生年・データ元）、血統（父・母・母父の名前）、通算成績（出走数・勝利数・連対率・複勝率）、そしてレース別の出走履歴（日付・レース・着順・タイム・上がり・人気）を一覧できる。

**Why this priority**: 出走馬の背景を一目で把握できることが、人間の意思決定支援の中核。レース詳細にすでに馬名があり、そこからの自然な深掘り導線になる。これ単体で「馬を調べられる」価値が成立する。

**Independent Test**: 任意のレース詳細で馬名をクリック→馬ページに遷移し、(a) 基本情報と血統名が表示される、(b) 通算成績が DB 実績から算出されて表示される、(c) レース別履歴が新しい順に並ぶ、ことを確認すれば独立して検証できる。

**Acceptance Scenarios**:

1. **Given** あるレース詳細を開いている, **When** 出走馬の馬名をクリックする, **Then** その馬のプロフィールページ（`/horses/{horse_id}`）に遷移する。
2. **Given** 過去に複数走の実績がある馬のページ, **When** 表示する, **Then** 出走数・勝利数・連対率・複勝率が実績から算出され、レース別履歴が並ぶ。
3. **Given** 血統名が登録されている馬, **When** 表示する, **Then** 父・母・母父の名前が表示される（ID 欠如でも名前で表示）。
4. **Given** 実績のないデビュー前/新馬（履歴ゼロ）の馬, **When** 表示する, **Then** 通算成績は 0 件として明示され（Unknown と 0 を区別）、履歴は空状態で示される（エラーにならない）。

---

### User Story 2 - 騎手プロフィールを見る (Priority: P2)

レース詳細を見ているユーザーが、騎乗している騎手の成績を知りたい。騎手名をクリックすると騎手プロフィールページに遷移し、基本情報（名前）と騎乗成績（騎乗数・勝率・連対率・複勝率）、最近の騎乗履歴（日付・レース・騎乗馬・着順）を一覧できる。

**Why this priority**: 騎手の調子・実績は予想判断の重要要素。馬ページと同じ「事実の深掘り」導線で、馬ページの仕組みを再利用できる。馬ページが無くても独立して価値があるが、規模・重要度から P2。

**Independent Test**: レース詳細で騎手名をクリック→騎手ページに遷移し、(a) 騎乗成績（騎乗数・勝率等）が実績から算出され、(b) 最近の騎乗履歴が並ぶ、ことを確認すれば独立して検証できる。

**Acceptance Scenarios**:

1. **Given** あるレース詳細を開いている, **When** 騎手名をクリックする, **Then** その騎手のプロフィールページ（`/jockeys/{jockey_id}`）に遷移する。
2. **Given** 騎乗実績のある騎手のページ, **When** 表示する, **Then** 騎乗数・勝率・連対率・複勝率が実績から算出され、最近の騎乗履歴が並ぶ。
3. **Given** 騎手 ID が紐づいていない（surrogate / 欠損）出走馬の行, **When** レース詳細を見る, **Then** その騎手名は非リンク（クリックできない）として表示され、壊れない。

---

### Edge Cases

- **未存在 ID**: 存在しない馬／騎手 ID へのアクセスは、明確な「見つかりません」状態（typed 404）で示し、500 にしない。
- **ID 欠損／surrogate**: 騎手 ID が無い（`nk:` surrogate や null）出走行は、リンクにせず名前のみ表示（遷移先が無いため）。馬/騎手 ID は固定フォーマットを持たない（JRA-VAN canonical も `nk:` surrogate もありうる）ため、未存在は一律 404 として扱う（形式 422 は設けない）。
- **履歴ゼロ**: 新馬・デビュー前など実績ゼロの馬／騎手は、成績 0 件・履歴空を明示（Unknown と 0 を区別）。
- **取消・除外馬**: レース別履歴で出走取消・競走除外は着順なしの状態として表示し、勝率等の母数の扱いを明確化（完走/出走の定義を一貫させる）。
- **大量履歴**: 長期の馬／騎手は履歴が多い → 表示件数の上限や区切りを設け、無制限に肥大しない。
- **同名異馬／同名騎手**: 表示は ID ベースで解決し、名前一致で別個体を混同しない。

## Requirements *(mandatory)*

### Functional Requirements

**馬プロフィール（US1）**

- **FR-001**: ユーザーはレース詳細から馬名をたどって、その馬のプロフィールページに遷移できる。
- **FR-002**: 馬プロフィールは基本情報（名前・性・生年・データ元）を表示する。
- **FR-003**: 馬プロフィールは血統（父・母・母父）を**名前**で表示する（ID が無くても名前で表示）。
- **FR-004**: 馬プロフィールは通算成績（出走数・勝利数・連対率・複勝率・平均着順）を、その馬の確定実績から算出して表示する。
- **FR-005**: 馬プロフィールはレース別の出走履歴（日付・レース・着順・タイム・上がり・人気）を新しい順に表示する。

**騎手プロフィール（US2）**

- **FR-006**: ユーザーはレース詳細から騎手名をたどって、その騎手のプロフィールページに遷移できる。
- **FR-007**: 騎手プロフィールは基本情報（名前）と騎乗成績（騎乗数・勝率・連対率・複勝率）を確定実績から算出して表示する。
- **FR-008**: 騎手プロフィールは最近の騎乗履歴（日付・レース・騎乗馬・着順）を表示する。

**遷移・契約**

- **FR-009**: レース詳細の出走表は、馬名・騎手名を（ID が解決できる行のみ）プロフィールへのリンクとして表示する。ID が無い行は非リンク表示にする。
- **FR-010**: 出走表のデータには、馬・騎手のプロフィールに遷移するための識別子（馬 ID・騎手 ID）が含まれていなければならない（現状の出走表は騎手 ID を返していないため、契約に追加する）。

**事実とモデル特徴量の区別（憲法 II/V）**

- **FR-011**: プロフィールに表示する成績は「確定実績の集計（事実）」であり、モデルの予測特徴量とは区別して提示しなければならない（混同させない）。
- **FR-012**: 本 feature はモデル特徴量・parquet 由来の値を表示しない（後続 feature に defer）。
- **FR-013**: プロフィールに表示する値はモデルの予測特徴量に流用してはならない（リーク境界、憲法 II）。
- **FR-014**: 実績ゼロは 0 件として明示し、データ欠如（Unknown）と 0 を区別する（憲法 IV の精神）。

**読み取り専用・境界（憲法 VI）**

- **FR-015**: 馬・騎手プロフィールの取得は読み取り専用でなければならない（既存の読み取り専用閲覧経路の原則を維持し、書き込み手段を増やさない）。
- **FR-016**: 未存在の馬／騎手 ID は「見つかりません」（404）として、未処理の 500 を出さずに型付きで応答する（馬/騎手 ID は固定フォーマットを持たないため形式 422 は設けない）。

### Key Entities *(include if feature involves data)*

- **馬（Horse）**: 既存。識別子・名前・性・生年・データ元、血統（父・母・母父の名前/ID）。プロフィールの主体。
- **騎手（Jockey）**: 既存。識別子・名前。プロフィールの主体。
- **出走（RaceHorse）**: 既存。レースと馬の関係、騎手 ID・人気・着前情報。馬/騎手の履歴・成績集計の元。
- **結果（RaceResult）**: 既存。着順・タイム・上がり等。成績集計（勝率・連対率・複勝率）と履歴の元。
- **馬プロフィール（Horse Profile）**: 上記から導出する表示用の集約（基本＋血統＋通算成績＋レース別履歴）。永続化しない（read 時に算出）。
- **騎手プロフィール（Jockey Profile）**: 同上（基本＋騎乗成績＋騎乗履歴）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーはレース詳細から1クリックで、出走馬の馬プロフィール（基本・血統・通算成績・履歴）に到達できる。
- **SC-002**: ユーザーはレース詳細から1クリックで、騎乗騎手のプロフィール（成績・騎乗履歴）に到達できる。
- **SC-003**: 馬／騎手の通算成績（出走数・勝率・連対率・複勝率）が、その個体の確定実績と一致する（手計算と突き合わせて検証できる）。
- **SC-004**: 実績ゼロの馬／騎手でもページが壊れず、0 件として表示される。
- **SC-005**: ID が解決できない出走行の馬名／騎手名は非リンクで表示され、リンク切れ・遷移エラーが起きない。
- **SC-006**: プロフィール取得経路に書き込み手段が増えていない（参照のみ）。
- **SC-007**: 未存在 ID で未処理の 500 が発生しない（型付き 404 で応答）。

## Assumptions

- 血統は **名前ベース**で表示する（実 DB で血統 ID はほぼ未投入。ID ベース結合は後続）。
- 成績集計は既存の結果テーブル（出走・結果）の集約で算出し、新たなスキーマ追加はしない（read のみ）。
- 「連対率＝2着以内率」「複勝率＝3着以内率」を採用し、母数は「出走（または完走）」で一貫定義する（spec/plan で確定）。
- レース別履歴・騎乗履歴は表示件数に上限を設ける（無制限に肥大させない。既定値は実装時に妥当値）。
- 馬／騎手 ID は既存の識別子契約（`id_mappings` 経由、surrogate `nk:` を含む）に従う。新たな名前一致結合は作らない。
- 本 feature は読み取り専用の閲覧 API（014）への additive な拡張として実装し、書き込み経路は増やさない。
- 特徴量・parquet 表示／調教師ページ／血統 ID 結合・3代血統・ニックス等は本 feature の対象外（deferred）。
