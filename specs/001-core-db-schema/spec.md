# Feature Specification: Core DB スキーマと基盤テーブル契約

**Feature Branch**: `001-core-db-schema`

**Created**: 2026-06-21

**Status**: Draft

**Input**: User description: "Core DB スキーマと基盤テーブル契約 (Core Database Schema & Foundational Table Contracts)"

## 概要

このプロジェクトの全データが乗る初期データスキーマと、その上の不変条件(制約)を定義する。
過去レース取込・特徴量生成・学習・評価・予測 serving・推奨が、後から破壊的なスキーマ変更を
せずに乗れる「テーブル契約」を最初に固定することが目的。憲法の原則 I (データ契約)・
V (再現性と監査)・VI (feature 分割規律) をデータ層で満たす。

本フィーチャーで作るのは、スキーマ・制約・マイグレーション・データ層の不変条件検証のみ。
実データの取込ロジック (JRA-VAN / netkeiba)、特徴量計算、学習、予測ロジックは別フィーチャー
であり、本スコープには含めない。ただし、それらが依存する最小契約 (キー・必須項目・監査項目・
状態項目) は本スキーマに含める。

「利用者」は人間エンドユーザーではなく下流コンポーネントである: 取込パイプライン、
特徴量 / 評価ハーネス、予測 serving、AdminFront のジョブ管理。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - コアレースデータを正規化して保持できる (Priority: P1)

取込パイプラインが、レース・馬・騎手・調教師・出走情報・結果を、一意キーと制約付きで
保存・upsert できる。これにより、以降のすべてのフィーチャーが参照する正規化済みデータ基盤
が成立する。

**Why this priority**: この基盤がなければ取込・特徴量・学習・予測のいずれも開始できない。
プロジェクト全体の前提となる最小スライスであり、単独で「データを保持できる」価値を提供する。

**Independent Test**: ダミーの 1 レース分 (レース・出走馬・結果) を投入し、すべての制約を満たして
保存されること、同一 `race_id` の再投入で行が重複せず最新値に更新されること、を検証する。

**Acceptance Scenarios**:

1. **Given** 空のデータベース、**When** 妥当な 1 レース分のレース・出走馬・結果を投入する、
   **Then** 制約違反なく保存され、`races` 1 行・`race_horses` 出走頭数分・`race_results` 完走頭数分
   が存在する。
2. **Given** 既に保存済みの `race_id`、**When** 同じ `race_id` で更新値を再投入する、
   **Then** 行数は増えず、対象項目が最新値に更新され、`updated_at` が更新される。
3. **Given** `race_horses` / `race_results` の行、**When** 同一 `(race_id, horse_id)` を重複投入する、
   **Then** 複合主キー制約により重複行が作られず upsert される。
4. **Given** 出走表に存在するが未登録の馬・騎手・調教師参照、**When** 出走情報を投入する、
   **Then** 参照整合性ルールに従って処理され (US3 の補完前提)、不整合が黙って混入しない。

---

### User Story 2 - リーク防止と学習ラベルがデータ層で保証される (Priority: P1)

評価ハーネスと特徴量生成が、リークなしに学習ラベルと時系列分割を作れるデータ土台を持つ。

**Why this priority**: 憲法の NON-NEGOTIABLE 原則 II (リーク防止)・III (評価先行) をデータ層で
担保する。ラベル導出と時系列分割が成立しないと、後続の評価ハーネス・学習が前提を満たせない。

**Independent Test**: 結果投入後にラベル導出クエリ (1着率 / 2着以内率 / 3着以内率 の教師) が
取消・除外・中止を正しく除外して整合した値を返すこと、`odds` の上書きで履歴行が増えないこと、
`race_date` による時系列分割クエリが「対象レースより前のみ」を表現できること、を検証する。

**Acceptance Scenarios**:

1. **Given** 完走・取消・除外・中止・同着を含む結果データ、**When** 教師ラベルを `finish_order`
   から導出する、**Then** 1着・2着以内・3着以内の各ラベルが整合し、取消・除外・中止は完走前提の
   集計から除外される。
2. **Given** 出走馬の `odds`、**When** 新しいオッズで上書きする、**Then** スナップショット履歴行は
   作られず、最新値のみが保持され `updated_at` が更新される。
3. **Given** 複数 `race_date` のレース、**When** ある基準日より前のレースのみを対象とする集計を行う、
   **Then** 基準日以降のレースが集計に混入しない (walk-forward 分割が表現可能)。
4. **Given** 過去成績のない馬、**When** その馬の過去成績由来データを参照する、
   **Then** 値は欠損 (Unknown / null) として区別され、`0` とは区別できる。

### User Story 3 - 異種ソースの ID を安全に対応付けられる + 取込が監査できる (Priority: P2)

JRA-VAN と netkeiba を推測結合せず、対応表経由でのみ突き合わせられ、取込ジョブの状態と
失敗理由を後から追える。

**Why this priority**: 憲法 I (ID は対応表経由でのみ横断利用) を担保する。取込が本格化する前に
対応表とジョブ監査の器を用意しないと、誤結合や原因不明の取込失敗が蓄積する。コアデータ保持
(US1/US2) があれば検証は始められるため P2。

**Independent Test**: 未対応 ID を含む取込シナリオで、推測結合されず未対応として記録され、
ジョブの状態 (成功 / 失敗 / 部分成功) とエラー理由が残ることを検証する。

**Acceptance Scenarios**:

1. **Given** JRA-VAN 側 ID と netkeiba 側 ID、**When** 対応表に対応が存在しない、
   **Then** 推測で結合されず、未対応として記録され、手動修正フローに乗せられる状態になる。
2. **Given** 同一正規エンティティへ複数ソース ID を対応付け、**When** 衝突 (同一ソース ID が
   複数の正規 ID に対応するなど) が発生する、**Then** 衝突が表現・検出でき、黙って上書きされない。
3. **Given** 取込ジョブの実行、**When** 一部成功・一部失敗で終了する、**Then** ジョブ種別・対象範囲・
   状態・エラー理由・実行時刻が記録され、後から追跡できる。

---

### User Story 4 - 予測・推奨を後付け破壊変更なしに保存できる (最小契約) (Priority: P2)

予測 serving と推奨が、結果を監査可能な形で保存できる最小テーブル契約を初期から持つ。

**Why this priority**: 憲法 VI (予測・推奨系テーブルの最小契約を初期 DB 設計に含める) と
V (再現性と監査) を担保する。後付けすると API と画面の作り直しになるため、実装は後回しでも
契約は今固定する。コアデータがないと意味を持たないため P2。

**Independent Test**: 1 件の予測実行と 1 件の推奨を投入し、後から「提示時点の判断根拠」を
監査項目だけで再構成できる (使用した市場オッズ・推定市場オッズ・疑似オッズ・疑似ROI・
計算時刻・ロジック版・モデル版) ことを検証する。

**Acceptance Scenarios**:

1. **Given** あるモデル版と対象レース、**When** 予測実行と馬別確率 (1着率 / 2着以内率 / 3着以内率)
   を保存する、**Then** 予測実行が `race_id`・モデル版・計算時刻・ロジック版に紐づいて保存される。
2. **Given** 保存済みの予測実行、**When** 推奨 (買い目候補) を保存する、**Then** 推奨に、計算時に
   使用した市場オッズ・推定市場オッズ・疑似オッズ・疑似ROI・計算時刻・ロジック版が保持される。
3. **Given** 最新オッズが後から上書きされた、**When** 過去の推奨を参照する、**Then** 推奨保存時の
   監査項目だけで提示時点の判断根拠を再構成でき、最新オッズの変化に影響されない。
4. **Given** 予測時点の特徴量、**When** 予測実行に紐づけて特徴量スナップショットを保存する、
   **Then** 後から同一入力で予測を再現できる参照が残る。

---

### Edge Cases

- 同一 `race_id` の再取込 (upsert) で、結果が後から確定するケース (出走情報のみ → 結果追加)。
- 出走表に存在するが `horses` / `jockeys` / `trainers` に未登録の参照 (補完前の整合性の扱い)。
- 取消・競走除外・競走中止・失格・同着の結果取込 (状態として区別し、疑似着順に変換しない)。
- 少頭数レース (出走頭数が極端に少ない場合のラベル・整合性)。
- 2006 年以前データの投入試行 (拒否され、取込・学習・評価に混入しない)。
- 海外馬・地方馬・未知血統の馬 (血統参照項目が欠損でも保存できる)。
- `race_results` が存在しない未来レース (出走情報のみ保持できる)。

## Requirements *(mandatory)*

### Functional Requirements

**コアレースデータ (US1)**

- **FR-001**: システムは `races`, `horses`, `jockeys`, `trainers`, `race_horses`, `race_results`
  のコアテーブルを保持しなければならない。初期項目は `docs/database.md` を正とする。
- **FR-002**: システムは `races.race_id` を主キーとし、`^[0-9]{12}$` を満たす値のみ受け入れ
  なければならない。
- **FR-003**: システムは `races.race_number` を 1〜12 の範囲に制約しなければならない。
- **FR-004**: システムは `race_horses` と `race_results` を `(race_id, horse_id)` の複合主キー
  としなければならない。
- **FR-005**: システムは同一 `race_id` の再取込を upsert として扱い、行を重複させず最新値に
  更新しなければならない。
- **FR-006**: システムは全テーブルに `created_at` と `updated_at` を持たせ、更新時に
  `updated_at` を更新しなければならない。

**リーク防止・ラベル (US2)**

- **FR-007**: システムは `race_results.finish_order` から学習ラベル (1着率 / 2着以内率 /
  3着以内率 の教師) を導出できる項目構成を持たなければならない。
- **FR-008**: システムは `races.race_date` を持ち、時系列 (walk-forward) 分割と「対象レースより
  前のみ集計」を表現できなければならない。
- **FR-009**: システムは出走馬オッズをスナップショット履歴として保存してはならない。最新値を
  上書きし、`updated_at` のみ保持しなければならない。
- **FR-010**: システムは過去成績の欠損を Unknown / null として表現でき、`0` と区別できなければ
  ならない。存在しない過去成績に `0` を代入してはならない。
- **FR-011**: システムは出走状態 (entry status) を `race_horses` 上で表現し、出走取消・競走除外を
  区別できなければならない。取消・除外は非出走であり、疑似着順 (最下位等) に変換してはならない。
- **FR-012**: システムは以下の正規状態コード体系を本フィーチャーで確定し、完走前提の集計から
  非完走・非出走を区別して除外できなければならない (論理状態を正とする。物理名・格納方式は plan
  で確定):
  - **出走状態 (race_horses, entry status)**: `started` (出走)、`cancelled` (出走取消)、
    `excluded` (競走除外)。`cancelled` / `excluded` は非出走であり `race_results` 行を持たない。
  - **完走状態 (race_results, result status)**: `finished` (完走)、`stopped` (競走中止)、
    `disqualified` (失格)。`stopped` / `disqualified` は出走済みだが完走前提の集計から除外する。
  - **同着 (dead heat)**: 独立した状態ではなく、同一レース内で複数馬が同一 `finish_order` を
    共有することで表現する。
  非出走 (`cancelled` / `excluded`) と非完走 (`stopped` / `disqualified`) は、平均着順・タイム等の
  完走前提特徴量から区別して除外できなければならない。

**ID 対応・取込監査 (US3)**

- **FR-013**: システムは馬・騎手・調教師について `(source, source_id)` と正規 id の対応を保持する
  `id_mappings` を持たなければならない。
- **FR-014**: システムは対応表に対応が存在しない異種ソース ID を、推測結合せず未対応として
  記録できなければならない。
- **FR-015**: システムは ID 対応の衝突 (同一ソース ID が複数正規 id に対応する等) を表現・検出でき、
  黙って上書きしてはならない。
- **FR-016**: システムは取込ジョブの種別・対象範囲・状態 (成功 / 失敗 / 部分成功)・エラー理由・
  実行時刻を記録する `ingestion_jobs` を持たなければならない。

**予測・推奨の最小契約 (US4)**

- **FR-017**: システムは学習済みモデルと校正器のバージョンを管理する `model_versions` を持たなければ
  ならない (識別子・作成時刻・採用状態・評価サマリ参照を含む)。
- **FR-018**: システムは予測実行単位を表す `prediction_runs` を持たなければならない
  (`race_id`・モデル版・計算時刻・ロジック版を含む)。
- **FR-019**: システムは予測実行ごとの馬別確率 (1着率 / 2着以内率 / 3着以内率) を保存する
  `race_predictions` を持たなければならない。
- **FR-020**: システムは予測時点の特徴量を固定する `feature_snapshots` を持ち、予測を再現できる
  参照を残さなければならない。
- **FR-021**: システムは推奨 (買い目候補と疑似ROI) を保存する `recommendations` を持ち、計算時に
  使用した市場オッズ・推定市場オッズ・疑似オッズ・疑似ROI・計算時刻・ロジック版を結果側に保持
  しなければならない (憲法 V の監査情報)。
- **FR-022**: システムは推定市場オッズ由来の値を実オッズ由来の値と区別できる形で保持しなければ
  ならない (疑似評価の明示)。
- **FR-023**: 券種別の結合確率・推定市場オッズの詳細項目は本スコープ外 (P0 未決) とし、本フィーチャー
  では確率を保持できる最小項目のみ定義する。これらの追加は後続フィーチャーで、既存契約への
  非破壊的拡張として行えなければならない。

**境界・運用**

- **FR-024**: システムは 2006 年以前のレースデータを取込・学習・評価の対象に混入させてはならない。
  この 2007-01-01 境界は取込レイヤのバリデーションで強制する。スキーマには `race_date` への
  ハードな日付制約 (CHECK 等) を入れない (境界は取込ポリシーであり、未来レースや将来の方針変更を
  妨げないため)。本フィーチャーのスコープでは、後続の取込フィーチャーがこの境界を強制できるよう
  `race_date` を保持し、検証 (SC-006) を提供する。
- **FR-025**: システムはスキーマ変更を冪等に適用・ロールバックできるマイグレーションとして提供
  しなければならない。

### Key Entities *(include if feature involves data)*

- **races**: レース単位の基本情報。`race_id` (12桁) が主キー。`race_date`・`race_number`・開催場・
  距離・馬場・天候・グレード・クラス・発走時刻・状態など。時系列分割の基準を提供する。
- **horses**: 馬の基本情報と血統参照 (sire / dam / damsire の id と名称)。血統参照は欠損可。
  `data_source` を持つ。
- **jockeys** / **trainers**: 騎手・調教師の基本情報。
- **race_horses**: レース出走馬。`(race_id, horse_id)` が主キー。枠・馬番・騎手・調教師・斤量・
  馬体重・増減・最新オッズ・人気・脚質・`cancelled` 状態など。発走前情報を保持。
- **race_results**: レース結果。`(race_id, horse_id)` が主キー。`finish_order`・タイム・着差・
  通過順・上がり 3F・失格や中止などの状態。学習ラベルの正となる。
- **id_mappings**: 馬・騎手・調教師の `(source, source_id)` ↔ 正規 id の対応。未対応・衝突を表現。
- **ingestion_jobs**: 取込ジョブの種別・対象範囲・状態・エラー理由・実行時刻。
- **model_versions**: 学習済みモデルと校正器のバージョン。採用状態・評価サマリ参照。
- **prediction_runs**: 予測実行単位。`race_id`・モデル版・計算時刻・ロジック版。
- **race_predictions**: 予測実行ごとの馬別確率 (1着率 / 2着以内率 / 3着以内率)。
- **feature_snapshots**: 予測時点の特徴量固定 (再現性のため)。
- **recommendations**: 買い目候補と疑似ROI。提示時点の監査情報 (使用オッズ・疑似オッズ・
  疑似ROI・計算時刻・ロジック版) を保持。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: コアの 6 テーブルが制約付きで作成され、ダミーデータで全制約 (`race_id` 形式、
  `race_number` 範囲、複合主キー、upsert) が検証でき、不正値がすべて拒否される。
- **SC-002**: `race_results` からのラベル導出 (1着率 / 2着以内率 / 3着以内率) が、取消・除外・中止を
  正しく除外して、整合した値を返す (確率ラベルの整合: 各馬 0/1、レース内 1着 1 頭・2着以内 2 頭・
  3着以内 3 頭の標準ケースで一致)。
- **SC-003**: 未対応の異種ソース ID が推測結合されず、`id_mappings` 上で未対応として記録され、
  ID 衝突が検出される。
- **SC-004**: 1 件の予測実行と推奨を保存し、最新オッズが後から上書きされても、推奨保存時の監査
  項目だけで提示時点の判断根拠を 100% 再構成できる。
- **SC-005**: マイグレーションが空 DB に対して冪等に適用でき、ロールバックで元の状態に戻せる。
- **SC-006**: 2006 年以前のレースデータの投入試行が拒否され、コアデータに混入しない。

## Assumptions

- データストアとマイグレーションツールは `aiuma` を踏襲する想定 (参考:
  `/Users/kuwatawaku/workspace/aiuma` の `0001_core_schema.py` /
  `0002_race_identity_constraints.py`)。具体採用は plan で確定する。
- 実データ投入・取込ロジック (JRA-VAN / netkeiba) は別フィーチャー。本スコープはスキーマ・制約・
  検証のみ。
- 券種別結合確率・推定市場オッズの詳細項目は P0 未決 (`docs/open-decisions.md`) のため後続
  フィーチャーへ deferred。本フィーチャーは非破壊拡張が可能な最小契約のみを固定する。
